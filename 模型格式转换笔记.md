## 核心概念对应

- 模型格式（PT → ONNX → TRT）：export 拆分 + patcher 修补 + trtexec 编译
- 部署（解封装 + 跑算子）：TrtEngineRunner 加载 engine → CUDA stream 异步推理
- 省显存（量化）：--fp16，权重从 4 字节 → 2 字节
- KV Cache（显存换时间）：maskmem_features/pos_enc 跨帧缓存，propagation 直接读取
- Chunk 机制（类比 MoE 只激活部分）：每次只推理 engine 容量内的物体数，用时间换空间

## 部署链路

```
PyTorch 模型 (.pt) → 拆分导出 (export) → ONNX 子模型 (.onnx)
→ 修补 (patcher) → 修好的 ONNX → TRT 编译 (trtexec --fp16)
→ TensorRT Engine (.engine) → TrtEngineRunner 异步推理 → 实时追踪结果

部署链路 = 拆分 → 导出 → 修补 → 编译 → 异步推理
```


# PyTorch checkpoint (.pt) 的本质：插槽名 → 数值的字典

## .pt 文件里存的是什么

`torch.save` / `torch.load` 保存的不是"一个完整的模型对象"，而是一个 **`OrderedDict[str, Tensor]`**——参数字符串名到权重张量的映射：

```python
ckpt = {
    "detector.backbone.language_backbone.encoder.transformer.resblocks.0.ln_1.weight": tensor(...),
    "detector.backbone.language_backbone.encoder.transformer.resblocks.0.ln_1.bias":   tensor(...),
    "detector.backbone.language_backbone.encoder.token_embedding.weight":               tensor(...),
    "detector.backbone.language_backbone.resizer.weight":                               tensor(...),
    ...
}
```

**checkpoint 本身不包含任何类的定义、`__init__` 逻辑或模型构建代码。** 它只是一个名称→数值的映射表。

## 模型结构怎么来：代码重建，然后填权重

模型结构（`nn.Module` 树）必须由代码显式构建。PyTorch 的 `nn.Module` 在 `__init__` 里搭好所有子模块后，自动维护一个 `state_dict()`：

```python
class VETextEncoder(nn.Module):
    def __init__(self):
        self.encoder = TextTransformer(...)   # 里面有 token_embedding, transformer.resblocks...
        self.resizer = nn.Linear(...)

model.state_dict().keys()
# → ["encoder.token_embedding.weight",
#    "encoder.transformer.resblocks.0.ln_1.weight",
#    "encoder.transformer.resblocks.0.ln_1.bias",
#    "resizer.weight", ...]
```

## load_state_dict 做了什么

逐 key 匹配，形状一致就填进去：

```python
# 伪代码
for key, ckpt_value in checkpoint.items():
    if key in model.state_dict():
        model_param = model.state_dict()[key]
        if ckpt_value.shape == model_param.shape:
            model_param.copy_(ckpt_value)  # 把数值填进对应的参数张量
```

在 SAM3 项目里（`model_builder.py:527-542`）：

```python
ckpt = torch.load(f, map_location="cpu", weights_only=True)
# checkpoint 里 key 是 "detector.xxx"，模型里直接是 "xxx"
# 所以先去掉 "detector." 前缀
sam3_image_ckpt = {
    k.replace("detector.", ""): v for k, v in ckpt.items() if "detector" in k
}
# 按名字匹配填入
model.load_state_dict(sam3_image_ckpt, strict=False)
```

## 关键前提

**代码构建的模型结构和 checkpoint 保存时用的结构必须一致**——参数名对不上就填不进去（`strict=False` 时跳过并打印 warning）。

所以项目的完整流程是：

```
① build_sam3_stream_model()  用代码搭好所有 nn.Module 子类（结构）
② model.load_state_dict(ckpt)  把 .pt 里的权重按名字填入（数值）
③ 得到完整的 PyTorch 模型 → 拆分成 5 个 ONNX 子图导出
```

**类比**：checkpoint 是一张"插槽名→数值"的表，代码预先搭好所有插槽（名字、形状、位置），`load_state_dict` 按名字找到对应插槽把值填进去。checkpoint 本身完全不包含"怎么搭这些插槽"的信息。

# PyTorch → ONNX → TensorRT 完整流程

```
PyTorch 模型 (.pt)
      ↓  torch.onnx.export()   记录计算图 + 权重
ONNX 模型 (.onnx)
      ↓  TensorRT builder      解析 + 优化（算子融合、内存复用）
TensorRT 引擎 (.engine)
      ↓  加载 + 推理
得到结果
```

## 第1步：PyTorch → ONNX（导出）

把 PyTorch 模型翻译成 ONNX 通用格式。TensorRT 不认识 PyTorch，只认识 ONNX。

```python
torch.onnx.export(
    text_encoder,                               # 要导出的模型
    (dummy_input,),                             # 示例输入，让模型跑一遍以记录计算图
    "text_encoder.onnx",                        # 输出文件路径
    input_names=["tokenized"],                  # 给输入节点起名
    output_names=["feat", "mask", "emb"],       # 给输出节点起名（顺序=forward 返回值顺序）
    dynamic_shapes={"tokenized": {0: prompt_dim}},  # 声明哪些维度运行时可变
    dynamo=True,                                # 使用 PyTorch 2.x Dynamo 导出路径
)
```

---

## 第2步：ONNX → TensorRT 引擎（构建）

TensorRT 读取 ONNX，做大量优化后生成 `.engine` 文件。

```python
builder = trt.Builder(logger)
network = builder.create_network()
parser = trt.OnnxParser(network, logger)
parser.parse_from_file("text_encoder.onnx")          # 解析 ONNX

# TensorRT 自动优化：算子融合、内存复用、选最优 kernel 等
engine = builder.build_serialized_network(network, config)

with open("text_encoder.engine", "wb") as f:
    f.write(engine)                                  # 保存引擎
```

> 例如模型里有 `矩阵乘法 + 加法 + ReLU`，TensorRT 会把三者融合成一个算子，减少两次显存读写。

## 第3步：加载引擎

```python
with open("text_encoder.engine", "rb") as f:
    engine_bytes = f.read()
```

### Engine vs Context

| 概念 | 作用 | 类比 |
|------|------|------|
| **Engine** | 存储权重、计算图、kernel，只读的"计算蓝图" | 编译好的程序 `.exe` |
| **Context** | 管理内存地址绑定、动态形状、CUDA 流，可变的"执行实例" | 进程的运行时环境 |
| **Index / Slot** | 张量在引擎中的槽位编号 | 数组下标 |
| `set_tensor_address` | 将 GPU 地址绑定到对应槽位 | 设置函数参数指针 |
| `execute_async_v3` | 启动 GPU 异步计算，按图依次 launch kernel | 调用函数 |

**Engine 是菜谱，Context 是一次做菜的过程**（锅的位置、火候、当前状态）。

### 索引（Index）是槽位编号，不是节点

引擎内部有固定的张量槽位表（类似数据库表结构），用户只能访问输入/输出槽位，内部中间张量不可见：

```
槽位  名称              角色      数据类型    形状
────────────────────────────────────────────────────
 0    tokenized         input     int64      [-1, 77]
 1    language_features output    float32    [-1, 512]
 2    language_mask     output    int32      [-1, 77]
 3    language_embeds   output    float32    [-1, 77, 768]
 4    hidden_0          internal  隐藏       ...
 5    attention_bias    internal  隐藏       ...
```

### Context 内部做了什么

```python
# ExecutionContext 的简化实现
class ExecutionContext:
    def __init__(self, engine):
        self.engine = engine
        self.num_slots = engine.num_slots
        self.tensor_addresses = [None] * self.num_slots
        self.binding_shapes = [None] * self.num_slots
        self.stream = None
        
        # ========== 关键：引擎分配内部临时槽位的显存 ==========
        # 构建时 engine 已经规划好：
        #   - 哪些槽位是用户手动分配的（is_user_managed）
        #   - 哪些槽位是引擎自动分配的（is_engine_managed）
        #   - 每个槽位需要的大小（slot.size）
        #   - 显存复用策略（哪些槽位共享同一块显存）
        
        # 遍历所有槽位，为引擎管理的槽位分配显存
        for slot_id, slot_info in enumerate(engine.slot_info):
            if slot_info.is_engine_managed:  # 内部临时槽位
                # CPU 调用 CUDA API 在 GPU 上分配显存
                size_bytes = slot_info.size
                # cudaMalloc 返回 GPU 显存地址
                gpu_address = cuda.malloc(size_bytes)  
                self.tensor_addresses[slot_id] = gpu_address
                # 此时显存已分配，但内容未初始化
                
        # 用户管理的槽位（输入/输出）初始为 None
        # 等待用户调用 set_tensor_address 绑定
        
    def set_tensor_address(self, index, address):
        """用户绑定输入/输出槽位的显存地址"""
        slot_info = self.engine.slot_info[index]
        if not slot_info.is_user_managed:
            raise RuntimeError(f"Slot {index} is engine managed, cannot set manually")
        self.tensor_addresses[index] = address
        
    def set_input_shape(self, index, shape):
        """设置动态形状（如果需要）"""
        self.binding_shapes[index] = shape
        # 如果形状变化导致需要重新分配显存...
        if self.engine.slot_info[index].is_dynamic:
            self._reallocate_if_needed(index, shape)
            
    def _reallocate_if_needed(self, index, new_shape):
        """动态形状时重新分配显存"""
        new_size = compute_size(new_shape)
        if new_size > self.engine.slot_info[index].allocated_size:
            # 重新分配更大的显存
            new_address = cuda.malloc(new_size)
            self.tensor_addresses[index] = new_address
            self.engine.slot_info[index].allocated_size = new_size
            
    def execute_async_v3(self, stream):
        # 1. 检查所有输入槽位是否有有效地址
        for input_slot in self.engine.input_slots:
            if self.tensor_addresses[input_slot] is None:
                raise RuntimeError(f"Input slot {input_slot} not bound")
        
        # 2. 检查所有输出槽位（如果是用户管理的，必须有地址）
        for output_slot in self.engine.output_slots:
            if self.tensor_addresses[output_slot] is None:
                raise RuntimeError(f"Output slot {output_slot} not bound")
        
        # 3. 按顺序执行所有 kernel
        for kernel in self.engine.kernels:
            # 收集这个 kernel 需要的所有地址
            args = []
            for slot in kernel.input_slots:
                addr = self.tensor_addresses[slot]
                if addr is None:
                    raise RuntimeError(f"Slot {slot} not allocated")
                args.append(addr)
            for slot in kernel.output_slots:
                addr = self.tensor_addresses[slot]
                if addr is None:
                    raise RuntimeError(f"Slot {slot} not allocated")
                args.append(addr)
            
            # 发射 kernel（包含硬编码的权重）
            # 注意：这里的地址有些来自用户（输入输出）
            #       有些来自引擎分配（内部临时）
            launch_kernel(kernel, args, stream)
        
        # 4. 立即返回，GPU 异步执行
    
    def __del__(self):
        """析构时释放引擎分配的显存"""
        for slot_id, addr in enumerate(self.tensor_addresses):
            slot_info = self.engine.slot_info[slot_id]
            if slot_info.is_engine_managed and addr is not None:
                cuda.free(addr)  # 释放 GPU 显存
```

## 第4步：喂数据 + 执行推理

```python
# 准备输入数据（GPU 上）
input_data = torch.tensor([[101, 2364, 102]], device='cuda')

# 准备输出容器（GPU 上的空张量，等待引擎填入结果）
feat = torch.empty(1, 512,       device='cuda')
mask = torch.empty(1, 77,        device='cuda')
emb  = torch.empty(1, 77, 768,   device='cuda')

# 告诉引擎数据在显存的哪个地址（零拷贝，不复制数据）
context.set_tensor_address(input_idx, input_data.data_ptr())
context.set_tensor_address(feat_idx,  feat.data_ptr())
context.set_tensor_address(mask_idx,  mask.data_ptr())
context.set_tensor_address(emb_idx,   emb.data_ptr())

# 异步执行
context.execute_async_v3(stream)
torch.cuda.synchronize()

# 此时 feat、mask、emb 已包含计算结果
```

### 推理时你需要提供什么 vs 不需要什么

引擎在构建时已经把**数据类型、计算图、固定维度**都烧死了。推理时你只需要给两样东西：

| 需要你提供 | 不需要你提供（引擎里已固化） |
|-----------|---------------------------|
| ✅ 显存地址（通过 `set_tensor_address`） | ❌ 数据类型——ONNX 导出时就定了，可以查询但不能改 |
| ✅ 实际形状（仅动态维度，通过 `set_input_shape`） | ❌ 固定维度的大小——如 `77`，已经写死在 engine 里 |
| | ❌ 内部计算图、融合后的算子、内存复用逻辑 |

```python
# 可以查询引擎里固化的信息，但不能改
input_idx = engine.get_binding_index("tokenized")
input_dtype = engine.get_binding_dtype(input_idx)   # → int64（只读）
input_shape = engine.get_binding_shape(input_idx)    # → (-1, 77)（动态维度显示 -1）
```

**类比**：数据类型和固定维度就像函数的参数类型签名——编译时定了 `int`，调用时就不能传 `float`。动态维度就是你声明了的可变参数。

### 静态形状 vs 动态形状：什么时候需要 set_input_shape

```
全程静态形状：
  ONNX 导出时没设 dynamic_shapes → dummy_input 的形状直接烧进 ONNX
  → TensorRT 构建时也烧进 engine
  → 推理时完全不需要 set_input_shape（形状已写死，没地方让你设）

有动态维度：
  ONNX 导出时设了 dynamic_shapes={...}
  → ONNX 保留"某维可变" → TensorRT 构建时接受这维可变
  → 推理时必须调 set_input_shape 告诉引擎"这次具体是多少"
```

| 场景 | 是否需要 `set_input_shape` | 原因 |
|------|--------------------------|------|
| 全程静态（没设 dynamic_shapes） | ❌ 不需要 | engine 里形状已烧死，不能设 |
| 有动态维度，每次形状可能变 | ✅ 必须（每次执行前） | 引擎需要知道本次的实际大小来验证地址空间 |
| 有动态维度，但其实永远不变 | ⚠️ 至少设一次 | context 初始状态不知道实际值 |

### 静态形状时怎么"指定"形状

不通过 API，而是在**创建 tensor 时就对上了**：

```python
# 全部静态——tokenized 在导出时固定为 [1, 77]
input_tensor = torch.tensor(token_ids, device='cuda')  # shape = [1, 77]
context.set_tensor_address("tokenized", input_tensor.data_ptr())
# 不需要 set_input_shape ——引擎早就知道 tokenized 只能是 [1, 77]
# tensor 形状不对的话，执行时会直接报错
```

### set_input_shape 做了什么

```python
context.set_input_shape("tokenized", (3, 77))
# 告诉 context："这次推理，tokenized 的实际形状是 [3, 77]"
```

context 内部做的事：
1. 更新 `binding_shapes` 表：slot 0 → `[3, 77]`
2. 验证绑定的显存地址空间 ≥ `3 * 77 * sizeof(int64)`
3. 如果形状变化导致内部 buffer 不够 → 自动重新分配

本质就是**给动态维度填具体值**。`dynamic_shapes` 声明了"长度可变"，`set_input_shape` 给了"这次长度具体是多少"。

### 完整推理循环（有动态维度）

```python
stream = torch.cuda.Stream()

for frame_idx, image in enumerate(frames):
    # 准备输入/输出张量（GPU 上）
    input_tensor = preprocess(image).to('cuda')   # shape = [1, 3, 640, 640]
    feat = torch.empty(1, 512, device='cuda')
    mask = torch.empty(1, 77, device='cuda')
    emb  = torch.empty(1, 77, 768, device='cuda')

    # ① 设置形状（仅动态维度需要）
    context.set_input_shape("tokenized", (input_tensor.shape[0], 77))

    # ② 绑定显存地址
    context.set_tensor_address("tokenized", input_tensor.data_ptr())
    context.set_tensor_address("feat", feat.data_ptr())
    context.set_tensor_address("mask", mask.data_ptr())
    context.set_tensor_address("emb", emb.data_ptr())

    # ③ 异步执行
    context.execute_async_v3(stream.cuda_stream)

    # ④ 同步（需要 CPU 读结果时）
    stream.synchronize()
    process_output(feat, mask, emb)  # 直接使用，无需 copy
```

### 构建时 vs 推理时：谁做了什么

| | 构建时 (Builder) | 推理时 (Runtime) |
|------|---------|-----------|
| **数据类型** | 从 ONNX 读取，固化 | 可以查询，不能修改 |
| **固定维度** | 从 ONNX 读取，固化 | 无需指定 |
| **动态维度范围** | 从 ONNX 读取，配置 profile | 每次调用前指定实际值 |
| **算子/计算图** | 融合、优化、固化 | 按图顺序发射 kernel |
| **显存** | 规划复用策略 | 实际分配 + 绑定地址 |

## execute 时硬件实际发生了什么

`set_tensor_address` 只是**登记地址**，真正的数据搬运发生在 `execute` 时：

```
显存（HBM）                      计算核心（SM / Tensor Cores）
──────────                      ──────────────────────────────

① 数据在显存里
   ┌──────────────┐
   │ 0x7f3a...    │  input_data 在这，等着被读
   └──────────────┘

                    ← ② execute 时，数据从显存搬到计算核心
                       ┌─────────────┐
                       │ SM          │
                       │ MatMul      │  ← Tensor Cores 算矩阵乘法
                       │ LayerNorm   │
                       │ Softmax     │
                       │ ...         │
                       └─────────────┘

   ┌──────────────┐
   │ 0x7f8b...    │  feat 结果写回这  ← ③ 算完的最终结果写回显存
   └──────────────┘
```

**类比**：

| 步骤 | 类比 |
|------|------|
| 显存（HBM） | 仓库 |
| `set_tensor_address` | 告诉工人："原料在 3 号货架，成品放 7 号货架" |
| compute cores（SM） | 工人 + 工作台 |
| `execute_async_v3` | 工人去 3 号货架取料 → 在工作台上加工 → 成品搬到 7 号货架 |
| 执行完毕 | 你直接从 `feat` tensor（7 号货架）里读结果 |

> `set_tensor_address` 不拷贝数据，只是登记地址。真正搬数据是 execute 时硬件**读显存 → SM 计算 → 写回显存**。

## 多个 Engine 怎么串联？（项目实战）

SAM3 把大模型拆成 **5 个独立 engine**，推理时手动串联：

```
┌──────────────┐
│ text_runner   │  ← 只跑一次，编码文本提示词
│ text_encoder  │
└──────┬───────┘
       │ language_features, language_mask （GPU 上的 tensor）
       ↓
┌──────────────┐
│ detector_     │  ← 每帧都跑，检测物体
│ frame_runner  │
└──────┬───────┘
       │ det_scores, det_masks, det_boxes     ← 检测结果
       │ trk_backbone_fpn_1/2/3              ← 中间特征，复用！
       ↓
┌──────────────┐
│ tracker_     │  ← 每帧每个目标都跑，跟踪物体
│ propagation  │
│ _runner      │
└──────┬───────┘
       │ tracker_low_res_masks, tracker_obj_ptr
       ↓
┌──────────────┐
│ tracker_     │  ← 更新记忆特征（存起来下帧喂回 propagation）
│ memory_update│
│ _runner      │
└──────────────┘
```

**具体代码（简化）**：

```python
# Step A: 文本编码（只跑一次，结果缓存）
text_out = self.text_runner.infer_device({"tokenized": tokenized})
# → {"language_features": tensor_gpu, "language_mask": tensor_gpu}

# Step B: 检测（每帧）
img_out = self.detector_frame_runner.infer_device({
    "image": image_tensor,
    "language_features_i": text_out["language_features"],  # 直接用上一步的输出！
    "language_mask_i": text_out["language_mask"],
})

# Step C: 跟踪传播（每帧每个目标组）
for group in self.tracker_groups:
    trk_out = self.tracker_propagation_runner.infer_device({
        "current_feat": img_out["trk_backbone_fpn_3"],    # 复用检测器中间特征
        "high_res_feat_1": img_out["trk_backbone_fpn_1"], # 避免重复计算 backbone
        "memory_prompt": prompts["memory"],                # 历史记忆
        ...
    })

# Step D: 更新记忆
    mem_out = self.tracker_memory_update_runner.infer_device({
        "current_feat": img_out["trk_backbone_fpn_3"],
        "low_res_masks": trk_out["tracker_low_res_masks"],
    })
    # mem_out["maskmem_features"] → 存到 group，下帧 Step C 用
```

**关键点**：
- 每个 engine 是**独立的** `infer_device()` 调用，各自有自己的 context
- engine 之间**零拷贝**：上一个的输出 tensor（GPU 上）直接当字典传给下一个
- backbone 中间特征（`trk_backbone_fpn_*`）**复用**给 tracker，避免重复算 backbone

# PyTorch → ONNX → TensorRT：转换与调用全链路

从三个视角拆解完整链路：**训练框架（PyTorch）→ 中间表示（ONNX）→ 推理引擎（TensorRT）**。

---

## 视角一：PyTorch（模块的世界）

PyTorch 的模型是**给人看的层次结构**，用 `nn.Module` 组织：

```python
class YOLOv11(nn.Module):
    def __init__(self):
        self.backbone = C2f()        # 模块（子组件）
        self.neck = C2f()            # 模块
        self.head = Detect()         # 模块
```

```python
class C2f(nn.Module):
    def __init__(self):
        self.bottlenecks = nn.ModuleList([Bottleneck() for _ in range(n)])
```

每个 `nn.Module` 是"乐高说明书上的一页"——帮助开发者理解、修改、调试。但硬件不关心这些分组。

---

## 视角二：ONNX（模块消失，只剩算子）

### export 做了什么

`torch.onnx.export()` 做了两种追踪方式：

| 方式 | 原理 | 特点 |
|------|------|------|
| **Tracing**（`torch.onnx.export` 默认） | 喂一个 dummy input，记录实际执行过的算子序列 | 看不到 `if/else` 分支 |
| **Dynamo**（PyTorch 2.x，`dynamo=True`） | 用 TorchDynamo 捕获 FX Graph，再翻译为 ONNX | 能处理动态控制流 |

不论哪种方式，输出的 ONNX 文件里**没有 Module 概念**，只有三种东西：

| ONNX 组成 | 含义 | 例子 |
|-----------|------|------|
| **Node**（节点） | 一个算子 | `Conv`、`Add`、`MatMul`、`Sigmoid` |
| **ValueInfo**（输入/输出） | 张量的元信息（形状、类型） | `float32[batch, 64, 32, 32]` |
| **Initializer**（初始值） | 权重/偏置的具体数值 | 卷积核的 float 数组 |

### 用代码验证

```python
import onnx

class Bottleneck(nn.Module):
    def __init__(self):
        super().__init__()
        self.conv = nn.Conv2d(64, 64, 3, padding=1)
        self.bn = nn.BatchNorm2d(64)
    def forward(self, x):
        return x + self.bn(self.conv(x))   # Conv → BN → Add(residual)

torch.onnx.export(Bottleneck(), (dummy_input,), "bottleneck.onnx")

onnx_model = onnx.load("bottleneck.onnx")
print([node.op_type for node in onnx_model.graph.node])
# 输出: ['Conv', 'BatchNormalization', 'Add']
# ❌ 没有 'Bottleneck'  —— 模块概念已消失
# ✅ 只有算子本身 —— Conv、BN、Add
```

### 为什么必须拆散

**因为下一步 TensorRT 要重新拼。** 拆得越散，重组的自由度越大：

```
PyTorch（说明书 + 积木） → 扔掉说明书，只留积木和连接关系
                            ↓
              ONNX（纯积木清单 + 连接图）
```

模块是人看的组织结构，算子才是机器跑的最小单元。

### ONNX 里模块的唯一痕迹：算子名

```python
# name 字段可能带来源信息（但也只是名字，不是结构！）
node.name = "/backbone/c2f.0/conv/Conv"
```

> 用 `pip install netron` 然后 `netron.start("model.onnx")` 可在浏览器中可视化整张计算图，确认只有算子节点。

## 视角三：TensorRT（优化 + 执行）

TensorRT 分两个阶段：**Build（构建/优化）** 和 **Runtime（加载/推理）**。

### 阶段一：Builder 构建 —— 把 ONNX 变成 .engine

Builder 不是简单地把 ONNX 算子一一翻译，而是做一套完整的**编译优化管线**：

```
ONNX 文件
   ↓  ① 图解析（Graph Parsing）
INetworkDefinition（网络定义）
   ↓  ② 图优化（Graph Optimization）
   │    ├── 垂直融合：Conv + BatchNorm + ReLU → 一个 kernel
   │    ├── 水平融合：并行的 ElementWise 合并
   │    ├── 消除冗余：去掉 Identity / Slice / Dropout
   │    └── 常量折叠：编译器已知的值直接算好
   ↓  ③ Myelin 图优化引擎
   │    ├── 跨层模式匹配
   │    ├── 静态内存调度（激活张量生命周期分析 → 复用）
   │    └── 硬件感知的内核选择
   ↓  ④ Kernel Auto-Tuning（Tactic Selection）
   │    ├── 每个算子枚举所有可行 kernel 实现（Winograd / GEMM / Implicit GEMM）
   │    ├── 在真实 GPU 上跑小规模 benchmark
   │    └── 选最快的那个
   ↓  ⑤ 序列化
.engine 文件（固化执行计划 + 权重 + 内存布局）
```

### 算子融合的本质

**不是为了"替换"算子，而是把多个节点合并成一个：**

```
ONNX 图（3 个节点）              TensorRT 图（1 个节点）
                                                       
  input                           input                  
    ↓                               ↓                    
  Conv  ← 节点1               ┌───────────┐             
    ↓                         │ FusedKernel│ ← 一个
  BN    ← 节点2   ──融合──→  │ (Conv+BN  │   kernel
    ↓                         │  +ReLU)   │             
  ReLU  ← 节点3               └───────────┘             
    ↓                               ↓                    
  output                          output                
                                                       
  原始节点已被合并为单一的融合 kernel                      
```

以 Conv+BN 融合为例，BN 参数被"吸收"进卷积权重，推理时完全跳过 BN 层：

```
fused_weight = conv_weight * (bn_gamma / sqrt(bn_var + eps))
fused_bias   = (conv_bias - bn_mean) * (bn_gamma / sqrt(bn_var + eps)) + bn_beta
```

一张 FP16 精度的 ResNet-50，融合后可带来 **1.5~2.5 倍**推理提速，主要来自：
- 减少 kernel launch 次数（每次 launch 有 μs 级固定开销）
- 中间结果不写回显存，走寄存器 / 共享内存传递
- 临时显存被复用或消除

### 静态内存调度

Builder 分析每个激活张量的"出生点到死亡线"：
- 张量 A 在第 3 层产出，在第 5 层消费完
- 张量 B 在第 6 层产出，在第 8 层消费完
- A 和 B 的生命周期不重叠 → **共享同一块显存**

推理时零 `malloc/free`，延迟波动极小。

### 阶段二：Runtime 加载与调用

```python
# ① 反序列化（CPU 读 .engine 文件）
with open("model.engine", "rb") as f:
    engine = runtime.deserialize_cuda_engine(f.read())

# ② 查看槽位表
for i in range(engine.num_io_tensors):
    print(engine.get_tensor_name(i))   # 按名字找到编号

# ③ 创建执行上下文
context = engine.create_execution_context()

# ④ 在 GPU 显存上分配输入/输出
input_data  = torch.randn(1, 3, 640, 640, device='cuda')  # 输入帧
output_data = torch.empty(1, 300, 4, device='cuda')       # 输出结果容器

# ⑤ 把显存地址填入槽位（零拷贝，只登记地址）
context.set_tensor_address("input",  input_data.data_ptr())
context.set_tensor_address("output", output_data.data_ptr())

# ⑥ 设置动态形状
context.set_input_shape("input", (1, 3, 640, 640))

# ⑦ 异步执行 —— CPU 发射所有 kernel 到 stream，立刻返回
stream = torch.cuda.Stream()
context.execute_async_v3(stream.cuda_stream)
# CPU 不阻塞，可以干别的事

# ⑧ 需要结果时同步
stream.synchronize()
# output_data 现在包含推理结果
```

### execute_async_v3 的内部行为

**用户只调用一次 `enqueue`，TensorRT Runtime 在内部批量发射所有 kernel。**

```
用户代码:    context.execute_async_v3(stream)    ← 只调一次
                │
                ▼
TensorRT     CPU 逐个 launch kernel 到 CUDA stream（封装在 enqueue 内部，对用户透明）
Runtime:     发k1 → 发k2 → 发k3 → ... → 发kN → 返回（干别的事）
GPU:               ↓ 队列排队（严格按发射顺序执行）
               取k1→跑→ 取k2→跑→ 取k3→跑→ ... → 取kN→跑
```

图是编译好的**固定序列**，没有 if/else 分支，所以 Runtime 不需要看中间结果——一次性全部发射。

CUDA stream 保证 GPU 端严格按发射顺序执行，kernel2 不会跑到 kernel1 前面。

同一个 stream 上，TensorRT kernel 跑完后如果后面跟着 PyTorch 算子，不需要手动 sync——CUDA stream 的有序性天然保证后续算子等到前面完成。只有需要把数据**从 GPU 读到 CPU** 时才需要 `stream.synchronize()` + `cudaMemcpy`。

**三层视角总结：**

| 层次 | 谁在发 kernel？ | 说明 |
|------|----------------|------|
| 🟢 用户代码层 | ❌ 不是你一个个发 | 你只调用一次 `enqueue` |
| 🟡 TensorRT Runtime 层 | ✔️ 内部批量 launch | 封装在 `enqueue` 内部，对你透明 |
| 🔴 CUDA 底层 | ✔️ kernel launch 本质仍是 CPU 发起 | 除非用了 CUDA Graph（见下文） |

### CUDA Graph：连 CPU launch 都省掉

TensorRT 在某些情况下会用 **CUDA Graph capture**，把一整串 kernel launch 捕获成**一次 graph launch**：

```
没用 CUDA Graph:  CPU → launch k1 → launch k2 → launch k3 → ... → launch kN
用了 CUDA Graph:  CPU → 一次 graph launch → GPU 自己按图执行所有 kernel
```

效果：CPU 开销更低，kernel launch 不再逐个触发。这是目前 NVIDIA GPU 上最极端的 CPU 减负手段。

### 多流并行（TensorRT 10.x+）

现代 TensorRT 支持在多个 stream 上并行发 kernel：

| Stream 类型 | 作用 |
|-------------|------|
| **Main stream** | `execute_async_v3(stream)` 传入的流，顺序执行 |
| **Auxiliary streams** | `context.set_aux_streams(...)` 设置，并行执行无依赖的 kernel |

TensorRT 自动管理跨 stream 同步：执行开始时辅助流等主流，结束时主流等辅助流。

## 完整转换流程：保留 vs 丢失

| 阶段 | 计算图 | 模块 | 算子 | 输入/输出命名 | 中间节点命名 |
|------|--------|------|------|---------------|--------------|
| **PyTorch** | ✅ 动态图 | ✅ 存在 | ✅ 基础算子 | ❌ 无 | ❌ 无 |
| **↓ ONNX 导出** | ✅ 静态图 | ❌ 消失 | ✅ 映射后算子 | ✅ 必须设置 | ⚠️ 自动生成 |
| **↓ TensorRT 构建** | ✅ 优化图 | ❌ 不存在 | ✅ 融合/替换后算子 | ✅ 继承保留 | ⚠️ 可能重命名 |

### 第 1 步：PyTorch → ONNX

```python
torch.onnx.export(

```

| ✅ 计算图 | 动态 → 静态（固定下来） |

| ✅ 算子 | `Conv2d` → `Conv`，`BatchNorm2d` → `BatchNormalization` |
| ✅ 输入/输出 | 有了名字（你指定的） |
| ⚠️ 中间节点 | 自动生成名字（`Conv_23`, `Add_5`...） |

### 第 2 步：ONNX → TensorRT

```python
parser.parse_from_file("model.onnx")
builder.build_serialized_network(network, config)
```

| 变化 | 说明 |
|------|------|
| ✅ 计算图 | 优化（层融合、常量折叠、拓扑排序） |
| ❌ 模块 | 依然不存在 |
| ✅ 算子 | `Conv`+`BN`+`SiLU` → `CudnnConvolution`（融合成 1 个） |
| ✅ 输入/输出命名 | 继承 ONNX 的名字 |
| ⚠️ 中间节点 | 可能被重命名或删除（融合后消失了） |

### 算子融合：命名的变化

```
ONNX 中的 3 个算子：
    [Conv] → [BatchNormalization] → [SiLU]
       ↓            ↓                   ↓
    Conv_0      BN_1                Sigmoid_2

TensorRT 融合后：
    [CudnnConvolution]  ← 1 个算子替代 3 个
       ↓
    conv_bn_silu_0      ← 新名字（旧名字消失）
```

### 命名传递链

```python
# PyTorch 导出时
input_names=["images"]

# ONNX 文件中
<graph>
  <input name="images" .../>   # ← 保存

# TensorRT 读取后

```

### 你只需要记住

| 你需要注意的 | 你不需要关心的 |
|--------------|----------------|
| ✅ 输入节点名字（推理时用到） | ❌ 中间节点名字 |
| ✅ 输出节点名字（推理时用到） | ❌ 模块叫什么 |
| ✅ 输入输出形状 | ❌ 算子怎么融合的 |
| ✅ 数据类型（FP16/FP32） | ❌ ONNX 自动生成的临时名字 |

**一句话总结：PT → ONNX：模块消失，算子映射，必须起输入输出名字；ONNX → TRT：算子融合/替换，输入输出名字保留不变。**

# 深入：Engine、Context 与 GPU 硬件的协作

## 谁在哪一侧？

```
CPU 侧（编排者）                          GPU 侧（执行者）
─────────────────                        ─────────────────
Engine 对象（计算图 + 槽位表）            显存（VRAM / HBM）——存权重和中间数据
ExecutionContext（地址映射表）             CUDA Cores —— 通用计算
Python 列表 tensor_addresses             Tensor Cores —— 矩阵乘加
cudaMalloc / cudaMemcpy 调用             CUDA Stream —— 命令队列
launch_kernel 发射命令
```

**CPU 负责"编排"，GPU 负责"执行"。** CPU 构建计算图、分配显存、填地址表、发射 kernel 命令，然后立刻返回。GPU 从 stream 队列取命令，自己去显存读数据、算、写结果。

## 计算图是 CPU 概念，不是 Tensor Cores

**Tensor Cores 是 GPU 上的专用硬件单元**，只做矩阵乘加（D = A × B + C），速度快。**计算图是 CPU 内存中的逻辑结构**——描述"先跑哪个 kernel，后跑哪个 kernel，数据从哪到哪"。

两者关系：图中的某些 kernel（卷积、全连接、Attention）内部会调用 Tensor Cores，但图本身是调度框架，不是硬件。

## 一个 kernel 内部：CUDA Cores + Tensor Cores 协同

```
┌──────────────────────────────────────┐
│ 一个卷积 kernel 的执行                │
│                                      │
│ CUDA Cores:                          │
│  - 数据搬运、im2col 展开             │
│  - ReLU / Add / LayerNorm            │
│  - 归约、reshape                     │
│                                      │
│ Tensor Cores:                        │
│  - 矩阵乘法（im2col 后的 matmul）     │
│  - Multi-head Attention 的 Q×K、V×... │
│  - 全连接层                          │
└──────────────────────────────────────┘
```

不是二选一，而是一个 kernel 内部混合使用。TensorRT 编译时自动决定哪些算子走 Tensor Cores（通常 FP16/INT8 精度下的矩阵运算）。

## 槽位 vs 中间张量

引擎的槽位表**只暴露输入/输出**，中间结果不可见：

```
槽位编号  名称           角色       显存
─────────────────────────────────────────
  0      tokenized      input     ← 用户分配，填入地址
  1      feat           output    ← 用户分配，填入地址
  ...    hidden_0       internal  ← TensorRT 内部 workspace，自动管理
  ...    attention_bias  internal  ← 自动管理，用户看不到
```

**中间张量存在 engine 预分配的一个大 workspace 里，算完即弃**，用户不需要为它们手动分配显存或绑定地址。

## tensor_addresses 列表在 CPU 上，存的是 GPU 门牌号

```python
self.tensor_addresses = [None] * num_slots   # 这是 Python 列表，在 CPU 内存
```

列表本身在 CPU 内存，每个元素是一个 **GPU 显存地址的数值**（device pointer，如 `0x7f3a00000000`），由 `cudaMalloc` 返回：

```
CPU 内存                                  GPU 显存
                                            
tensor_addresses (Python list)              
┌──────────┬──────────┬──────────┐          
│ 槽位0    │ 槽位1    │ 槽位2    │          
│ 0x7f3a00 │ 0x7f3a40 │ None     │          
└────┬─────┴────┬─────┴──────────┘          
     │          │                            
     ▼          ▼                            
  ┌──────┐  ┌──────┐                         
  │输入帧 │  │输出框 │                         
  │数据   │  │数据   │                         
  └──────┘  └──────┘                         
   GPU 显存   GPU 显存                        
```

**槽位是 CPU 侧的"注册表"，存的是 GPU 那边的门牌号。** GPU 执行 kernel 时通过地址值直接去显存对应位置读/写数据。

## 完整执行流程（一张图）

```
1. CPU 加载 engine 文件 → 反序列化 → 得到计算图 + 槽位表

2. for 每个输入/输出槽位:
     cudaMalloc(显存, size) → 返回地址 0x7f3a0000
     context.set_tensor_address(slot, 0x7f3a0000)  # CPU 填表

3. context.execute_async_v3(stream)   ← 用户只调一次
     TensorRT Runtime 内部遍历 engine.kernels:
       for kernel in kernels:
         args = [tensor_addresses[slot] for slot in kernel.inputs]
         args += [tensor_addresses[slot] for slot in kernel.outputs]
         launch_kernel(kernel, args, stream)  # 扔进 CUDA stream 队列
     CPU 立刻返回（不等 GPU）──→ CPU 可以干别的事

4. GPU 从 stream 队列取命令
     ┌──────────────────────────────────┐
     │ 根据地址去显存读输入              │
     │ CUDA Cores + Tensor Cores 计算   │
     │ 结果写回显存（输出槽位指向的位置）  │
     └──────────────────────────────────┘

5. 如需 CPU 读取结果:
     cudaStreamSynchronize(stream)  # 阻塞，等 GPU 干完
     cudaMemcpy(host, device, ...DeviceToHost)  # 显存 → 内存
     # 现在 CPU 端 numpy/tensor 有结果了
```

## 结果如何回到 CPU

默认情况，**结果留在 GPU 显存，CPU 拿不到**。要让 CPU 读到结果需要两步：

1. **同步**：`cudaStreamSynchronize(stream)` —— CPU 阻塞，等 GPU 算完
2. **拷贝**：`cudaMemcpy(host_ptr, device_ptr, size, cudaMemcpyDeviceToHost)` —— 显存 → 内存

很多实时推理场景根本不做这一步——GPU 输出的结果直接送给渲染/编码管线，全程在 GPU 侧闭环。

## cv2 绘制帧 = 结果必然回到 CPU

cv2（OpenCV）操作的是 **numpy 数组**，numpy 默认在 CPU 内存。所以如果推理 + cv2 画框 + NVENC 编码：

```
NVDEC 解码 → GPU 显存 (帧)
               ↓ cudaMemcpy D2H     ← PCIe 搬运 ①
          CPU 内存 (numpy 数组)
               ↓ cv2.rectangle / putText
          CPU 内存 (画好的帧)
               ↓ cudaMemcpy H2D     ← PCIe 搬运 ②
          GPU 显存 (待编码)
               ↓
NVENC 编码 → 码流
```

**两次 PCIe 往返**。想避免的话就在 GPU 侧完成所有操作（如 GStreamer/DeepStream 的 GPU 叠加插件，或 cupy 直接在显存上操作），让整条链路不离开显存。不过非极端场景（30fps、1080p）这个开销通常感知不到。

## 多 Context 共享一个 Engine

同一个 engine 可以创建多个 ExecutionContext，各自绑定不同的显存地址和 CUDA stream，实现在不同流上并行处理多路数据：

```python
engine = load_engine("model.engine")

# 线程 1
ctx1 = engine.create_execution_context()
ctx1.set_tensor_address(0, input1_ptr)
ctx1.execute_async_v3(stream1)

# 线程 2
ctx2 = engine.create_execution_context()
ctx2.set_tensor_address(0, input2_ptr)
ctx2.execute_async_v3(stream2)
# 两个推理并行，共享同一份权重（显存只占一份）
```

| 概念 | 作用 | 类比 |
|------|------|------|
| **Engine** | 计算图 + 槽位表 + 权重，只读 | 编译好的程序 `.exe` |
| **Context** | 每次执行的地址绑定 + stream，可变 | 进程的运行时环境 |
| **Slot** | 张量在引擎中的编号 | 数组下标 |
| `set_tensor_address` | 把 GPU 显存地址绑定到槽位 | 告诉工人原料在哪、成品放哪 |
| `execute_async_v3` | CPU 遍历图发射所有 kernel 到 stream，异步返回 | 按下开关，机器运转 |
| CUDA Cores | 通用计算（ReLU、归约、搬运） | 普通工人 |
| Tensor Cores | 矩阵乘加（卷积、全连接、Attention） | 专用焊工 |

# 模型格式与推理引擎：谁调用谁，怎么调用

不同模型格式需要对应的推理引擎才能执行。就像 `.docx` 需要 Word、`.pdf` 需要 PDF 阅读器。

## 模型格式 ↔ 推理引擎对照表

| 模型格式 | 原生推理引擎 | 替代方案 | 特点 |
|----------|-------------|----------|------|
| PyTorch (.pt/.pth) | PyTorch (torch) | ONNX Runtime, TensorRT | 训练框架，推理较慢 |
| TensorFlow (.pb/.h5) | TensorFlow | TFLite, ONNX Runtime | 生态完善 |
| ONNX (.onnx) | ONNX Runtime | TensorRT, OpenVINO, TVM | 通用中间格式 |
| TensorRT (.engine) | TensorRT | 无 | NVIDIA 专用 |
| GGUF (.gguf) | llama.cpp | Ollama, LM Studio | LLM 量化格式 |
| Core ML (.mlmodel) | Core ML | 无 | Apple 生态 |
| TFLite (.tflite) | TensorFlow Lite | 无 | 移动端/嵌入式 |
| OpenVINO (.xml/.bin) | OpenVINO | 无 | Intel 生态 |

**实际例子：YOLOv11**

```bash
best.pt       # PyTorch 格式 → 只能用 PyTorch 推理
best.onnx     # ONNX 格式 → ONNX Runtime / TensorRT 都能用
best.engine   # TensorRT 格式 → 只能 TensorRT 用

# 推理速度对比
PyTorch:      100 FPS  (慢)
ONNX Runtime: 150 FPS  (中)
TensorRT:     300 FPS  (快)
```

**总结**：模型格式 = 存储方式，推理引擎 = 执行器。部署时根据硬件选格式和引擎。

## 调用链路：从最顶层到硬件

### 总的层次结构

```
用户的 Python / CLI 代码
    ↓
框架/库（Transformers, vLLM, Ollama...）
    ↓
计算核心（PyTorch C++, llama.cpp, ONNX Runtime...）
    ↓
底层加速库（cuDNN / cuBLAS / MKL / Metal）
    ↓
硬件驱动（CUDA / MPS / CPU Driver）
    ↓
CPU / GPU 执行
```

**每一层都是对下一层的封装。** 写 `model(x)` 而不写 CUDA kernel，这就是抽象的价值。

### 主流框架调用链路对照表

| 框架/库 | 用户入口 | 模型抽象层 | 计算核心层 | 硬件接口层 | 硬件 |
|---------|----------|-----------|-----------|-----------|------|
| **Transformers** | `model.generate()` | HF 模型定义（基于 `torch.nn.Module`） | PyTorch（动态计算图、自动微分） | CUDA/MPS/CPU | GPU/CPU |
| **PyTorch** | `output = model(x)` | `torch.nn` 模块（Module, Functional） | PyTorch C++ Runtime（ATen 张量库） | CUDA(cuDNN/cuBLAS)/MPS/CPU(MKL) | GPU/CPU |
| **vLLM** | `llm.generate(prompts)` | vLLM 模型封装（包装 HF 模型） | PagedAttention 内核 + PyTorch基础算子 + 自定义CUDA算子 | CUDA(cuDNN/cuBLAS)/ROCm(AMD) | GPU(NVIDIA/AMD) |
| **Ollama** | `ollama run` / REST API | GGUF 模型解析器（llama.cpp 格式） | llama.cpp（纯 C++ 优化算子） | CUDA/Metal(Apple)/CPU(OpenBLAS) | GPU/CPU |
| **llama.cpp** | `./main -m model.gguf` | GGUF 模型加载器 | 自研 C++ Kernels（MMQ, FlashAttention...） | CUDA/Metal/CPU(AVX2/AVX512) | GPU/CPU |
| **ONNX Runtime** | `session.Run()` | ONNX 模型图（标准 IR） | 执行提供者(EP)架构，分发到不同后端 | CUDA(TensorRT EP)/CPU/OpenVINO/DirectML | GPU/CPU |
| **TensorRT** | `context.execute()` | TensorRT 引擎（优化后的 IR） | 自研 CUDA Kernels（极致算子融合） | CUDA / DLA（NV 边缘设备） | GPU(NVIDIA) |

### 核心区别

| 角色 | 框架 | 一句话 |
|------|------|--------|
| **高级工具箱** | Transformers | 提供统一接口去调用底层发动机（通常是 PyTorch），自身不带发动机 |
| **通用发动机** | PyTorch | 所有上层工具的"地基"，提供动态计算和自动微分 |
| **高性能特制引擎** | vLLM | 在 PyTorch 地基上，针对大模型推理的内存瓶颈和并发问题深度改造 |
| **轻量一体机** | Ollama / llama.cpp | 纯 C++ 从头构建的精简推理后端，不依赖 PyTorch，开箱即用 |
| **性能压榨专家** | TensorRT | 为 NVIDIA GPU 生成极致优化的执行引擎，延迟最低 |
| **跨平台适配器** | ONNX Runtime | "一套模型，多处运行"，通过 EP 切换硬件后端 |

## 为什么 Transformers 下载的模型 vLLM 可以直接用？

因为 vLLM 和 Transformers **共享 PyTorch 作为底层计算引擎**，所以能直接加载同一份模型格式。

### 机制：vLLM 的 Transformers 建模后端

```
HuggingFace 模型仓库（config.json + .safetensors）
    ↓
    ├── vLLM 专用模型实现（如 LlamaForCausalLM 的 vLLM 版本）
    │      → 高度优化的 PagedAttention + 自定义 CUDA 内核
    │
    └── 没有专用实现？→ 走 Transformers 后端
           → 加载 HuggingFace 的 torch.nn.Module，用 PyTorch 算子跑
           → 官方承诺性能差异在 1%~5% 以内
```

### 为什么不需要额外下载格式？

| 你关心的问题 | 结论 |
|-------------|------|
| vLLM 和 Transformers 用同一种格式吗？ | 是的，都原生支持 HuggingFace 的 Transformers 格式 |
| 需要转换格式吗？ | 不需要，直接用 |
| 性能有损失吗？ | 几乎没有，官方承诺差异 1~5% |
| vLLM 能加载 GGUF 吗？ | 不能，vLLM 只支持 HF 格式 |

**一句话：同一份 `config.json` + `.safetensors` 权重文件，Transformers 和 vLLM 都能读，因为背后都是 PyTorch。**

## PyTorch 本身就是对硬件的封装

PyTorch 不是魔法——它就是对 CUDA/cuDNN/MKL 的层层封装，最终所有调用都落到硬件：

```
你的 Python 代码
    ↓
PyTorch Python API（类型检查、自动微分、图追踪）
    ↓
PyTorch C++ Runtime（ATen 张量库——统一调用底层库）
    ↓
cuDNN（卷积优化）/ cuBLAS（矩阵乘法）/ MKL（CPU 数学库）
    ↓
CUDA 驱动 / CPU 指令
    ↓
GPU SM / CPU Core 执行
```

**用一个卷积操作来看：**

```python
conv = nn.Conv2d(3, 64, 3)
output = conv(input)

# 背后调用链：
# nn.Conv2d → PyTorch C++ conv2d → cudnnConvolutionForward() → GPU
# 如果 input 在 CPU 上，则走 MKL 或 oneDNN
```

验证：同一个 `torch.matmul`，取决于 tensor 在哪个设备：

```python
x_cpu @ x_cpu   # → 调用 MKL（CPU 数学库）
x_gpu @ x_gpu   # → 调用 cuBLAS（NVIDIA 线性代数库）
# 同样的 PyTorch 代码，不同的底层！
```

**总结：**

```
PyTorch    = 对 CUDA/cuDNN/MKL 的封装
CUDA/cuDNN = 对 GPU 硬件的封装
MKL        = 对 CPU 指令集的封装
最终所有调用都落到硬件执行
```

**PyTorch 的价值**：让你写 `model(x)` 就能运行，而不是手写 CUDA kernel。
**TensorRT 的价值**：在 PyTorch 的基础上，进一步优化算子组合和内存访问模式。
**vLLM 的价值**：在 PyTorch 的基础上，针对 LLM 推理的 KV Cache 内存管理做深度定制。

# 最终总结：生成引擎 vs 调用引擎

## 引擎文件里有什么

Builder 把 ONNX 编译成 `.engine`，里面只有两个核心结构：

```
.engine 文件
├── 计算图（Kernel 序列）
│   └── 每个 Kernel = 固化了权重的优化算子
│       ├── 矩阵乘法（权重已嵌入 kernel 代码，运行时只读不改）
│       ├── 融合算子（Conv+BN+ReLU → 一个 kernel）
│       └── 没有"层"和"模块"概念，纯粹的操作序列
│
└── 槽位表（I/O 绑定信息）
    每个槽位记录了：
    ├── 名称：如 "tokenized"、"feat"
    ├── 编号：0, 1, 2, ...
    ├── 数据类型：int64 / float32 / ...
    ├── 形状：[-1, 77]（-1 表示动态维度）
    ├── 角色：input / output / internal（内部中间张量）
    └── 管理模式：user_managed（用户分配显存）/ engine_managed（引擎自动分配）
```

**一句话**：引擎 = 固化了权重的计算图 + 描述输入/输出接口的槽位表。

---

## 阶段一：生成引擎（Build Time）

```
ONNX → TensorRT Builder → .engine
```

| Builder 做的事 | 说明 |
|---------------|------|
| **算子融合** | Conv+BN+ReLU 合并为一个 kernel，权重吸收入融合算子 |
| **常量折叠** | 编译时已知的值直接算好，不留在运行时 |
| **自动调优** | 每个算子枚举多种 kernel 实现，在 GPU 上实测，选最快的 |
| **槽位表生成** | 从 ONNX 的 I/O 生成槽位，内部中间张量也分配槽位 |
| **内存规划** | 分析张量生命周期，不重叠的共享显存，固化内存映射 |
| **序列化** | 打包计算图 + 权重 + 槽位表 → 写入 .engine 文件 |

### 构建后定死的东西

- 数据类型——`float32` 不能换成 `float16`（除非构建时指定）
- 固定维度——`77` 写死在引擎里，不能改
- 计算图顺序——kernels 的执行顺序已确定，无分支
- 权重——已嵌入 kernel，推理时只读

### 构建后留有弹性的东西

- 动态维度（如 prompt 条数）——可以变，但有 min/opt/max 范围
- 显存地址——槽位里存的只是门牌号，运行时才知道具体指向哪块显存

---

## 阶段二：调用引擎（Runtime）

```
.engine → deserialize → context → 填槽位 → execute
```

### 第一步：创建 Context 时（内部自动分配）

```python
context = engine.create_execution_context()
```

此时引擎遍历所有槽位，对于 `engine_managed` 的槽位（内部中间张量），**自动调用 `cudaMalloc` 在 GPU 显存上分配空间**，并填入 `tensor_addresses` 表。这些槽位用户看不到也摸不着。

### 第二步：推理前（用户手动分配）

```python
# ① 按形状分配显存
input_tensor  = torch.randn(3, 77, device='cuda')   # 有数据
output_tensor = torch.empty(3, 512, device='cuda')  # 空容器，等 GPU 写入

# ② 如果有动态维度，告诉 context 这一次的实际大小
context.set_input_shape("tokenized", (3, 77))

# ③ 把显存地址绑到槽位
context.set_tensor_address("tokenized", input_tensor.data_ptr())
context.set_tensor_address("feat",      output_tensor.data_ptr())
```

用户只需要管输入/输出。输入的显存放的是**真实数据**，输出的显存只需要**空的容器**（占好坑，等 GPU 填）。

### 第三步：执行（一次 enqueue，Runtime 内部发射所有 kernel）

```python
context.execute_async_v3(stream.cuda_stream)   # 用户只调一次
```

用户只调了一次 `enqueue`。TensorRT Runtime 在内部做的是：
```
for kernel in engine.kernels:                          # 按预先定好的顺序
    args = [tensor_addresses[slot] for slot in kernel.inputs]   # 输入地址
    args += [tensor_addresses[slot] for slot in kernel.outputs] # 输出地址
    launch_kernel(kernel, args, stream)                # 扔进 CUDA stream，立刻返回
return  # 不等 GPU
```

GPU 做的是：按 stream 队列顺序取 kernel → 去槽位地址读数据 → CUDA Cores + Tensor Cores 计算 → 写回槽位地址。

> **关键认知**：用户代码只调了一次 `enqueue`，不是一个个发 kernel。逐个 launch 是 TensorRT Runtime 在 `enqueue` 内部完成的事，对用户透明。如果用了 CUDA Graph，连逐个 launch 都省了——一次 graph launch 替代一串 kernel launch。

### Kernel 之间的依赖关系

**没有运行时依赖解析。** 依赖是在构建时就处理好的：

```
在同一个 stream 上，kernel 严格按发射顺序执行——CUDA stream 的有序性天然保证：
  kernel2 绝对不会跳到 kernel1 前面
  kernel3 读的地址如果在 kernel2 写入 → 一定读到的是 kernel2 算完的结果
```

如果 TensorRT 10.x+ 用了辅助 stream，依赖管理由引擎在构建时自动插入同步点，用户感知不到。

---

## 一页纸总览

```
┌──────────────────────────────────────────────────────────────────┐
│                      引擎文件 (.engine)                           │
│                                                                  │
│  计算图                     槽位表                                │
│  ┌────────────────┐    ┌────┬──────┬──────┬──────┬──────────┐   │
│  │ [Kernel1]      │    │编号│ 名称  │ 类型  │ 形状  │ 管理模式   │   │
│  │  ↓             │    ├────┼──────┼──────┼──────┼──────────┤   │
│  │ [Kernel2]      │    │ 0  │input  │int64 │[-1,77]│用户管理   │   │
│  │  ↓             │    │ 1  │output │f32   │[-1,512│用户管理   │   │
│  │ [Kernel3]      │    │ 2  │hidden │f32   │[1,256]│引擎管理   │   │
│  │  ↓             │    └────┴──────┴──────┴──────┴──────────┘   │
│  │ ...            │                                              │
│  │ (固化权重+顺序) │                                              │
│  └────────────────┘                                              │
└──────────────────────────────────────────────────────────────────┘
         │                                        │
         │ Build Time（构建时）                     │ Runtime（运行时）
         │ 只做一次                                 │ 每次推理都做
         ▼                                        ▼
  算子融合、常量折叠                        ┌──────────────────────┐
  权重嵌入、自动调优                        │ 用户管：cudaMalloc    │
  内存规划、序列化                          │ + 填数据 + 绑地址     │
                                           │                      │
                                           │ 引擎管：自动分配      │
                                           │ CPU 发射 → GPU 运算  │
                                           └──────────────────────┘
```

**本质**：引擎是一个**预编译好的、带固定权重和固定执行计划的 GPU 程序**。Context 是这个程序的**一次运行实例**。用户只调一次 `enqueue`，TensorRT Runtime 在内部逐条发射 kernel 指令到 CUDA stream，GPU 负责在显存上取数据、算、写结果。如果用了 CUDA Graph，连逐条发射都省了——一次 graph launch 替代全部 kernel launch。
