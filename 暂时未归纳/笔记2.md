## 1. VLM 架构的模块化分解

**所属：** 架构设计

**描述：** VLM 可以分解为三个独立阶段——embed_tokens 将文本索引转为词向量，vision_tower（独立 ViT）将图像编码为视觉特征并通过 mm_projector 投影到与文本相同的 hidden_size，两者拼接后送入共享的 28 层 Transformer Decoder，最后经唯一的 lm_head 输出 logits。整个流程是：`input_ids → embed_tokens → text_embeds` + `images → vision_tower → mm_projector → image_embeds` → 拼接 → Decoder → lm_head。

**冲突：** 文本和图像来自完全不同的模态，特征分布差异大，直接拼接可能导致模型偏向某一模态或对齐不充分；视觉塔和 LLM 是分别预训练的，直接拼在一起训练时容易发生灾难性遗忘。

**解决方法：**
- （1）使用 projector（Linear/MLP）将视觉特征映射到与文本相同的 hidden_size，作为模态对齐的桥梁
- （2）分阶段训练：先冻结 LLM 只训练 projector 做对齐，再逐步解冻更多参数联合训练
- （3）使用多尺度视觉特征（主视觉塔 + 辅助视觉塔 DaViT）提供更丰富的视觉信息，减少信息瓶颈

## 2. 3D 多模态感知系统的完整运行链路

**所属：** 架构设计 / 多模态融合

**输入模态：**
- 多视角相机（6~12个）：纹理、语义、颜色丰富，但无真实深度
- LiDAR 点云：真实几何结构、精确距离，但稀疏无语义
- 核心矛盾：相机语义强但无 3D，雷达几何强但无语义 → 融合本质是把语义和几何对齐

**完整 Pipeline（五步）：**

| 步骤 | 做什么 | 子空间视角 |
|------|--------|------------|
| Step 1：各模态独立编码 | Camera → CNN/ViT → image feature；LiDAR → PointNet/SparseConv → point feature | 不同模态映射到各自特征子空间，避免梯度冲突 |
| Step 2：坐标对齐 | LiDAR 点投影到图像，或图像 feature lift 到 3D（Lift-Splat）→ 统一到 BEV 空间 | 人工规定的共享坐标系 |
| Step 3：融合 | BEV feature → Transformer/Attention 融合（中期融合为主流） | Attention = 在子空间之间"对齐+交换信息"，实现受控耦合 |
| Step 4：统一表示 | 融合后得到 H_shared（BEV feature） | 共享子空间 |
| Step 5：多任务输出 | Detection / Segmentation / Tracking / Planning 各 head：y_i = w_i · h | 不同任务在共享空间上沿不同方向读出 |

**三种融合策略对比：**
- Early fusion（直接拼接）：模态冲突严重，对齐困难
- Late fusion（各自预测再融合）：信息利用不充分
- **中期融合（主流）**：BEV feature → Transformer/Attention，在子空间之间做受控信息交换

**关键洞察：**
- **BEV = 人工设计的"共享子空间"**——不是让模型自己学，而是直接规定所有信息必须在这个空间对齐，大幅降低梯度冲突和学习难度
- 多任务 loss 不冲突的原因：各任务 head 在共享空间的不同方向读出
- 新增模态（如 radar）≈ 插入一个新的低维子空间（类似 LoRA 的思路）
- Multi-head attention 中不同 head = 不同融合方式（如一个对齐 geometry，一个对齐 semantic）
- **Fusion 的真正瓶颈是"坐标对齐误差"，而非模型能力**

**一句话总结：** 3D 多模态系统不是把所有输入丢进一个模型，而是：先在各自子空间编码 → 在统一空间（BEV）对齐 → 用 attention 做受控融合 → 在共享表示上做多任务读出。

**可继续深入的方向：**
- BEVFormer / PETR 本质是"query 在 3D 空间主动取信息"
- 纯视觉 3D（完全不用 LiDAR）如何做出来
- 为什么 fusion 瓶颈是坐标对齐误差而非模型能力

## 3. BEVFormer / 多视角感知 / Transformer 检测 核心链路总结

---

# 🧠 一、BEV 的本质是什么？

> **BEV（Bird's Eye View）= 把多视角图像统一到"车体坐标系的俯视空间表示"**

---

## ✅ 输入不是一张图，而是：

```text
多相机图像（6~8个）
```

---

## ✅ 输出是：

```text
BEV 特征图：(B, H, W, C)
例如：(1, 200×200, 256)
```

---

👉 含义：

* 每个格子 = 地面一个位置 (x, y)
* 每个格子有一个 feature 向量

---

# 🧱 二、BEV 是怎么从多视角来的？

---

## 核心思想：

> **不是拼图，而是"投影 + attention 融合"**

---

## 流程（非常重要）：

```text
多相机图像
    ↓
Backbone / FPN（提特征）
    ↓
BEV queries（空间网格）
    ↓
投影到各个相机
    ↓
采样图像特征
    ↓
融合（attention）
    ↓
BEV feature
```

---

## 🧠 关键点：

👉 BEV query：

```text
固定空间位置 (x, y)
```

👉 它做的是：

```text
"我这个位置，在图像里长什么样？"
```

---

# 🧠 三、BEV query 在"找什么证据"？

---

## ✔ 空间证据（SpatialCrossAttention）

```text
来自多视角图像的 feature
```

---

## ✔ 时间证据（TemporalSelfAttention）

```text
来自 prev_bev（历史帧）
```

---

👉 合起来：

```text
空间 + 时间融合
```

---

---

# ⏱ 四、时序建模（warp + TSA）

---

## 为什么需要？

车在动：

```text
t-1 ≠ t
```

---

## ✅ 解决：warp（坐标对齐）

```text
prev_bev → 对齐到当前坐标系
```

---

## 然后：

```text
当前 BEV ↔ 历史 BEV
```

---

👉 得到：

```text
变化 = 运动信息
```

---

## 🧠 本质：

> **通过对比同一空间位置在不同时间的特征变化，学习运动（motion）**

---

---

# 🧠 五、Decoder 在干嘛？（检测核心）

---

## 输入：

```text
BEV feature（地图）
+
object queries（900个）
```

---

## object query 是什么？

```text
一个"候选物体"
```

---

---

## Decoder 做三件事：

### 1️⃣ Self-Attention

```text
query ↔ query（去重、交流）
```

---

### 2️⃣ Cross-Attention（核心）

```text
query → BEV
```

👉 就是：

> ✔ **找物体证据**

---

### 3️⃣ FFN

```text
更新 query 表示
```

---

👉 多层之后：

```text
query → 真正的物体表示
```

---

---

# 🎯 六、Detection Head 输出什么？

---

每个 query 输出：

```text
类别（车、人）
+
3D box：
(cx, cy, cz, w, l, h, θ, vx, vy)
```

---

👉 所以：

```text
BEVFormer = 直接做 3D 检测
```

---

---

# 🧠 七、FCOS3D Head（辅助分支）

---

## 它做什么？

```text
在 image 上做检测（2D + 3D）
```

---

## 作用：

```text
辅助监督 backbone
```

---

👉 因为：

```text
image → BEV 路太长
```

---

---

# 🧠 八、Batch（B）是什么？

---

## 推理时：

```text
B = 1
```

---

## 训练时：

```text
B = 2~8
```

---

👉 注意：

```text
时间 ≠ batch
```

---

时间通过：

```text
prev_bev
```

传递

# 🧠 九、BEV 有没有高度？

---

## ✔ 有三种情况：

---

### 1️⃣ BEVFormer（你现在这个）

```text
隐式高度（3D采样 → 压成2D）
```

---

### 2️⃣ BEV + height bins

```text
(x, y, z_bins)
```

---

### 3️⃣ 3D voxel / occupancy

```text
(x, y, z)
```

---

👉 趋势：

```text
BEV → 3D world modeling
```

---

---

# 🧠 十、整个系统一句话串起来（最重要）

---

```text
多相机图像
    ↓
BEV Encoder（用 BEV query 构建地图）
    ↓
BEV feature（统一空间表示）
    ↓
Decoder（用 object query 找物体）
    ↓
Detection Head（输出 3D box）
```

---

---

# 🎯 最终总结（建议记住这一句）

> **BEVFormer 的核心是：用 BEV query 在统一空间中融合多视角和时序信息构建地图，再用 object query 在这个地图上提取证据完成3D检测。**
