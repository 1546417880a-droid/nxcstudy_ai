# VLM 多模态大模型笔记

---

## 1. VLM 架构的模块化分解

**所属：** 架构设计

**描述：** VLM 可以分解为三个独立阶段——embed_tokens 将文本索引转为词向量，vision_tower（独立 ViT）将图像编码为视觉特征并通过 mm_projector 投影到与文本相同的 hidden_size，两者拼接后送入共享的 28 层 Transformer Decoder，最后经唯一的 lm_head 输出 logits。整个流程是：`input_ids → embed_tokens → text_embeds` + `images → vision_tower → mm_projector → image_embeds` → 拼接 → Decoder → lm_head。

**冲突：** 文本和图像来自完全不同的模态，特征分布差异大，直接拼接可能导致模型偏向某一模态或对齐不充分；视觉塔和 LLM 是分别预训练的，直接拼在一起训练时容易发生灾难性遗忘。

**解决方法：**
- （1）使用 projector（Linear/MLP）将视觉特征映射到与文本相同的 hidden_size，作为模态对齐的桥梁
- （2）分阶段训练：先冻结 LLM 只训练 projector 做对齐，再逐步解冻更多参数联合训练
- （3）使用多尺度视觉特征（主视觉塔 + 辅助视觉塔 DaViT）提供更丰富的视觉信息，减少信息瓶颈

## 2. 高维向量空间的多任务信息分布

**所属：** 表征学习

**描述：** 每个 token 经过 28 层 self-attention 后的 4096 维向量不只是它自己的表征，而是从序列中所有其他 token 拉取信息后的"压缩信息包"。一个图像 patch token 的向量里同时携带了语义信息（与文本 token 的 cross-attention）、空间位置（ViT 位置编码 + 邻域 patch 交互）、尺度信息（多尺度特征）、以及物体间关系（全局 self-attention）。

**为什么一个 lm_head 就能同时输出语义和坐标？** 不是不同的头选择不同子空间，而是同一个头，Decoder 喂给它不同的 hidden_state。lm_head 的权重 W 是固定的（4096×152064），但每个 token 位置的 h 是变化的——Decoder 通过 self-attention 学到：当前要输出语义时就把语义信息压进 h，要输出坐标数字时就把空间信息压进 h。W 中不同 token 对应的行在训练中自然分化：坐标数字 token 的行学会响应空间模式，语义词 token 的行学会响应语义模式。所以是：

```
当前要输出"猫" → Decoder 把语义信息压进 h → W 中"猫"对应的行高响应
当前要输出"100" → Decoder 把空间信息压进 h → W 中"100"对应的行高响应
```

**关键：一个 W，不同的 h，就能读出不同的东西，不需要多个独立任务头。**

**冲突：** 不同任务的信息挤在同一个 hidden_state 的 4096 维里，会相互干扰。比如语义信息和空间位置信息纠缠在一起，lm_head 难以区分当前该"读语义"还是"读坐标"；或者某些任务分配到的有效维度太少（如空间信息只占了少量维度），导致定位精度不够；还有信息分布不均匀的问题——某些子空间过载，另一些维度浪费。

**解决方法：**
- （1）增大 hidden_size（如 4096→8192），给各任务足够维度，减少对子空间的竞争
- （2）使用特殊标记分隔任务上下文（如 `<|box_start|>(100,200)<|box_end|>`），让 Decoder 在生成不同任务 token 时把不同类型的信息压进 h 的不同方向，降低干扰
- （3）分阶段/多任务训练：先学语义理解，再引入定位任务，让子空间逐步分化而不是一次性抢占
- （4）解耦特征分支：不同模态用不同的 projector（如主投影器处理全局视觉、辅助投影器处理区域特征），在进入 Decoder 前就把信息分流

## 3. 类继承关系与 MRO（方法解析顺序）

**所属：** 代码架构

**描述：** 模型有两层继承链 —— CausalLM 层（顶层包装）和 Model 层（backbone），两个 `Meta` 基类各司其职：

```
CausalLM 层 (顶层包装):
OmChatQwen25VLForCausalLM
  ├── Qwen2_5_VLForConditionalGeneration  (MRO 先)
  └── OmChatMetaForCausalLM               (MRO 后)
       │  定义 get_vision_tower(), get_vision_tower_aux(),
       │       encode_videos() 等方法
       │  所有方法都通过 self.get_model() 委托
       │
       └── self.model = OmChatQwen25VLModel(config)

Model 层 (backbone):
OmChatQwen25VLModel
  ├── OmChatMetaModel                     (MRO 先)
  │    持有 vision_tower, mm_projector, vision_tower_aux,
  │         object_vp_extractor(HFRE), mm_projector_aux
  └── Qwen2_5_VLModel                     (MRO 后)
       核心 transformer decoder
```

**关键：** `OmChatMetaForCausalLM` 和 `OmChatMetaModel` 在两个不同的继承链上，不直接交互。子类 `OmChatQwen25VLForCausalLM` 通过重写 `get_model()` 返回 `self.model` 来连接两层。

**调用链示例 —— `self.get_vision_tower()`：**
1. 调用 `OmChatMetaForCausalLM.get_vision_tower()`（子类没重写，走父类）
2. 内部调用 `self.get_model()`，此时是多态调用，走到子类的 `OmChatQwen25VLForCausalLM.get_model()`（行 245）
3. 返回 `self.model`（`OmChatQwen25VLModel` 实例）
4. 再调用 `.get_vision_tower()`，此时走到 `OmChatMetaModel.get_vision_tower()`（行 44）
5. 返回真正的 `self.vision_tower` 对象

## 4. OD（目标检测）完整 Pipeline

**所属：** Pipeline 流程

### 4.1 数据预处理：三种 token 类型

以模板 `"Please detect {} in this image. Answer the question with object indexes."` 为例：

| 类型 | 例子 | token ID | 从哪里获取向量 |
|---|---|---|---|
| 语言 token | `Please`, `detect`, `<region0>` | 正常词表 ID | `embed_tokens` 词嵌入矩阵查表 |
| 图像占位符 | `<image>` | `-200` (IMAGE_TOKEN_INDEX) | 主视觉塔 → mm_projector |
| 区域占位符 | `<regionfeat>` | `-300` (DEFAULT_REGION_INDEX) | 辅助塔+主塔 → HFRE → mm_projector_aux |

**训练时序列结构：**
```
[151644] [-200] [文本token...] [-300] [-300] [-300]
   ↑       ↑                     ↑     ↑     ↑
  bos   图像占位符            bbox0  bbox1  bbox2
                             特征   特征   特征
```

**推理时序列结构（多了 `<regionN>` 文本标签）：**
```
<region0><regionfeat><region1><regionfeat><region2><regionfeat>
```

### 4.2 `<regionN>` 的双重作用

`<region0>` 是一个真实 token，会经过 `embed_tokens` 变成词嵌入向量。模型通过两重信息定位"这是第几个候选区域"：

- **隐式**：`-300` 在序列中的出现顺序 = `bbox_list` 的遍历顺序（严格按序对应）
- **显式**：`<region0>`, `<region1>` 文本 token 经 embed_tokens 变成嵌入向量，让模型直接读出索引号

### 4.3 区域特征提取：模型不自己找位置

OD 流程是两阶段的：

1. **外部检测器（SAM3/UPN）生成候选框** → 模型只做"验证"，判断哪些候选框包含目标物体
2. **HFRE 用 ROIAlign 提取每个框的特征**：

```
DaViT 4层特征                      Qwen ViT 4层特征
    │                                    │
    ▼ concat                             ▼ concat
  [1, 3840, H, W]                     [1, 5120, H, W]
    │                                    │
    ▼ ROIAlign(box, 0.25, 7)            ▼ ROIAlign(vt_boxes, 1/14, 7)
  [N, 3840, 7, 7]                     [N, 5120, 7, 7]
    │                                    │
    ▼ mean(dim=(2,3))                    ▼ mean(dim=(2,3))
  [N, 3840]                           [N, 5120]
    │                                    │
    └────────── concat ──────────────────┘
                    │
                    ▼  nn.Linear → [N, 4096]
                    ▼  替换 input_ids 中的 -300
```

### 4.4 bbox 坐标变换链

| 阶段 | 坐标空间 | 函数 |
|---|---|---|
| 原始数据 | 像素坐标 [x1,y1,x2,y2] | COCO 格式 |
| 图像 resize | 按比例缩放 | `resize_shortest_edge_images_and_bboxes` |
| 辅助塔预处理 | 768×768 | `adjust_bbox` |
| 主塔坐标再缩放 | 动态分辨率 | `encode_regions` 中 scale_height/scale_width |
| HFRE 内部 | 归一化到 [0,1]，转中心点+宽高 | ROIAlign 前 |

## 5. 位置编码系统

**所属：** 模型架构 / 位置编码

**描述：** 模型中有两套**完全独立**的位置编码系统，服务于不同模块。

### 5.1 Vision Tower（图像编码器）—— 2D RoPE

**核心函数**：`rot_pos_emb`（modeling_qwen2_5_vl.py:674-718）

- 为每个 image patch 生成 (h, w) 二维坐标
- (h, w) 作为索引查一张共享的 1D 频率表
- 高度和宽度方向共享同一套频率基底，仅索引值不同
- `spatial_merge_size=2` 时做 reshape+permute，让将被 merge 的 2x2 patch 获得相邻位置
- **实现类**：`Qwen2_5_VisionRotaryEmbedding`
- **应用函数**：`apply_rotary_pos_emb_vision`

```
一张 20x30 个 patch 的图:
  patch(i,j) 的 Q,K 被施加:
    cos(h_i, w_j) * Q  +  sin(h_i, w_j) * rotate_half(Q)
这样 ViT attention 就知道 patch 之间的 2D 空间关系。
```

### 5.2 LLM 解码器 —— 3D MRoPE（Multimodal RoPE）

**核心函数**：`get_rope_index`（modeling_qwen2_5_vl.py:1906-2045）

`position_ids` 形状为 `[3, batch, seq_len]`：

| 通道 | 文本 token | 图像 patch token |
|---|---|---|
| 通道0 (T) | = 线性位置 | = 0（单帧图像） |
| 通道1 (H) | = 线性位置 | = row index |
| 通道2 (W) | = 线性位置 | = col index |

文本 token 三个通道值完全相同 → 退化为普通 1D RoPE。

**`mrope_section` 切分 head_dim：**

```
mrope_section = [24, 20, 20] x2 = [48, 40, 40]
head_dim(128) = [0..47]时间 + [48..87]高度 + [88..127]宽度
cos/sin 最终排列: [T0, H0, W0, T1, H1, W1, ...]
```

**为什么文本三个通道值相同？**
因为 `mrope_section` 是固定的——无论 token 是文本还是图像，head_dim 都按同一比例切分给 T/H/W。文本没有真正的 2D 位置，三通道填相同值，效果上等价于普通 1D RoPE，实现上避免了对文本和图像分别走不同路径。

- **实现类**：`Qwen2_5_VLRotaryEmbedding`
- **应用函数**：`apply_multimodal_rotary_pos_emb`

### 5.3 两套位置编码对比

| | Vision Tower | LLM Decoder |
|---|---|---|
| 类型 | 2D RoPE | 3D MRoPE |
| 位置维度 | (h, w) 共享频率表 | (T, H, W) 三通道独立频率 |
| head_dim 切分 | 不切分 | 按 `mrope_section` 切三段 |
| 文本处理 | N/A | 三通道填相同值 |
| 实现类 | `Qwen2_5_VisionRotaryEmbedding` | `Qwen2_5_VLRotaryEmbedding` |
| 位置 ID 来源 | `rot_pos_emb(grid_thw)` | `get_rope_index(input_ids, image_grid_thw, ...)` |

**关键不干扰机制：** Vision Tower 的 2D RoPE 在 ViT 内部就已消费完毕，输出给 LLM 的是不含位置信息的纯语义特征向量。LLM 拿到后再用 3D MRoPE 给整个多模态序列（文本+图像+区域）统一编码位置。

### 5.4 `rope_deltas` 机制

由于图像的 position_ids（含 2D 空间坐标）可能与序列长度不一致，在 autoregressive generation 的首步（prefill）计算 `rope_deltas = max(position_ids) - seq_len`，后续 steps 用它偏移新生成 token 的位置 ID。

## 7. 视觉 Token 位置信息的两种传递方式

**所属：** 架构设计 / 位置编码

### 7.1 问题背景

Vision Tower 把图像编码成 N 个 patch token 后，需要告诉 LLM"每个 token 在图像中的哪个位置"。这件事有两种完全不同的做法。

### 7.2 方法一：位置向量直接加到 token 里（加法融合）

```
图像 → 切 patches → 每个 patch 加上可学习的位置向量 → ViT 编码 → 输出给 LLM
```

```python
# 示意代码
patches = image_to_patches(img)        # [600, 768]  ← 20x30 个 patch
pos_embed = learnable_2d_pos(20, 30)   # [600, 768]  ← 可学习的 2D 位置编码表
patches = patches + pos_embed          # 位置信息"融"进了特征里
visual_tokens = vit(patches)           # ViT 处理
# 输出: [600, 768]，每个向量里混合了语义+位置，无法再拆开
```

**特点：**
- 位置信息隐含在特征向量中，无法显式读取
- LLM 不知道哪个 token 在哪个位置，只能从混合特征中"猜"
- 优点：简单、高效、训练稳定
- 缺点：LLM 难以做精确的空间定位（"第3行第5列有什么？"→ 它不知道哪个 token 对应那个位置）

**代表模型：** ViT、DeiT、CLIP、早期的 BLIP-2

### 7.3 方法二：语义和位置分离，LLM 用 MRoPE 重新编码

```
图像 → ViT(内部用 2D RoPE) → 纯语义 token  +  显式坐标 → 交给 LLM → LLM 用 MRoPE 统一编码位置
```

```python
# 示意代码
visual_tokens = vit_with_2d_rope(image)  # ViT 内部用 2D RoPE 处理空间关系
# 输出: 纯语义 token [600, 768]，位置已在 ViT 内部消费完毕，不残留在输出中

positions = [(row, col) for each patch]  # 显式坐标 [(0,0), (0,1), ..., (19,29)]

# LLM 侧：文本 token 用 1D 位置，图像 token 用 2D 坐标，统一走 MRoPE
llm_input = [text_0, text_1, img_patch_0, img_patch_1, ...]
llm_positions = [0, 1, (0,0), (0,1), ...]  # MRoPE: 文本退化为1D，图像用2D坐标
```

**特点：**
- ViT 输出的 token 是纯语义的，位置信息已剥离
- LLM 拿到显式的坐标元数据，可以按需用 MRoPE 动态编码
- 能精确回答空间问题："左上角的红色物体是什么？"
- 能泛化到训练时没见过的分辨率

**代表模型：** Qwen2-VL、Qwen2.5-VL、LLaVA-NeXT、InternVL2

### 7.4 直观类比

| | 方法一（加法融合） | 方法二（分离设计） |
|---|---|---|
| 类比 | 快递单上把地址写在货物标签里，取出货物时地址已贴好 | 货物和地址分开交付，快递员可以根据需要重新打包寻址 |
| 位置在哪 | 隐含在特征向量的数值里 | 显式坐标 + MRoPE 旋转编码 |
| LLM 能否修改位置理解 | 不能，位置已在特征中固定 | 能，根据任务动态重编码 |
| 空间定位能力 | 弱，需要从混合特征中"解码"位置 | 强，直接通过注意力对齐坐标 |

### 7.5 为什么新模型都转向方法二？

多模态对话越来越需要精确的空间定位能力：
- "图片左上角的红色物体是什么？"
- "把第二行第三列的物体圈出来"
- "第三个洞和第一个洞有什么不同？"

分离设计 + MRoPE 让 LLM 可以：看到每个视觉 token 的显式坐标 → 理解文本中的空间词汇 → 通过注意力自动对齐视觉区域和空间描述。

## 8. 位置编码演进史（从小白视角）

**所属：** 背景知识 / 位置编码

本节以零基础可读的方式梳理位置编码从文本到多模态的完整演进。

### 8.1 为什么需要位置编码？

Transformer 的 self-attention 本身是**位置无关**的——它只看 token 之间的内容相关性，不知道谁在前谁在后。如果没有位置编码，"我愛你" 和 "你愛我" 在模型眼中完全相同。所以必须给每个 token 注入位置信息。

### 8.2 第一代：绝对位置编码（Sinusoidal / Learned）

**做法：** 给序列的第 0, 1, 2, ... 个位置各分配一个固定的向量，直接加到 token embedding 上。

```
token_embedding[0] = word_embed("我") + pos_embed[0]
token_embedding[1] = word_embed("爱") + pos_embed[1]
token_embedding[2] = word_embed("你") + pos_embed[2]
```

- **Sinusoidal（正弦）**：用 sin/cos 函数生成，不需要学习
- **Learned（可学习）**：直接当参数训练

**优点：** 简单直观，原始 Transformer 就用这个
**缺点：** 只能处理训练时见过的长度（如 512），超出就"不认识"了

### 8.3 第二代：RoPE（旋转位置编码，2021）

**核心思想：** 不直接加位置向量，而是把 token 向量"旋转"一个角度，角度大小取决于位置。

```
RoPE(token, pos) = 把向量旋转 pos × θ 度
```

**关键性质：** 两个 token 做 attention 时，点积结果只取决于它们的**相对位置**（你在我前面 3 个位置），而不是绝对位置（你在第 5、我在第 8）。这跟人类阅读很像——我们关注的是"这个词在前面多远"，而不是它的绝对编号。

**优点：** 天然支持相对位置，长度外推能力强（训练 2K，推理可到 32K+）
**代表模型：** LLaMA、Qwen、ChatGLM 等几乎所有主流 LLM

### 8.4 RoPE 的原理（直觉版）

想象一个 2D 向量 `[x, y]`，把它旋转某个角度后，向量的长度不变，但方向变了：

```
旋转前: [3, 4]  长度=5
旋转 30° 后: [3*cos30 - 4*sin30, 3*sin30 + 4*cos30]  长度还是 5
```

RoPE 就是对高维向量（如 128 维）里的每一对相邻数字 `(dim0, dim1)`, `(dim2, dim3)`, ... 分别做 2D 旋转，每对的旋转角度不同（高频对旋转快，低频对旋转慢）。位置越远的 token 旋转角度越大，attention 点积就能感知到这个距离。

### 8.5 第三代：2D RoPE（图像专用）

**问题：** 图像 patch 按行列展平成 1D 序列后会丢失空间关系——同一列的相邻 patch 在序列中可能相隔几十个 token。

**做法：** 把 embedding 维度切成两半，一半编码高度，一半编码宽度：

```
RoPE_2D(patch, row=3, col=5) = 
  前半维度用 row=3 做 RoPE 旋转 + 后半维度用 col=5 做 RoPE 旋转
```

这样 ViT 的 self-attention 就能感知"这个 patch 在上方、那个在右边"。

### 8.6 第四代：MRoPE（多模态 RoPE，2024）

**问题：** 一个序列里同时有文本 token（1D 位置）、图像 token（2D 位置）、视频 token（3D 位置），怎么统一编码？

**做法（Qwen2-VL）：** 把 head_dim 切成三块，分别编码时间(T)、高度(H)、宽度(W)：

```
mrope_section = [16, 24, 24]  → 总 head_dim = 64
  前 16 维 ← 时间方向旋转
  中 24 维 ← 高度方向旋转
  后 24 维 ← 宽度方向旋转
```

| Token 类型 | T | H | W | 效果 |
|---|---|---|---|---|
| 文本 | 线性递增 | 线性递增 | 线性递增 | 退化为普通 1D RoPE |
| 图像 patch | 全为 0 | 行号 | 列号 | 2D 空间编码 |
| 视频 patch | 帧号 | 行号 | 列号 | 3D 时空编码 |

### 8.7 Qwen2.5-VL 的改进：T-RoPE 对齐绝对时间

Qwen2-VL 中视频帧的时间 ID 是简单递增的 `[0, 1, 2, 3, ...]`，不管视频是 1 FPS 还是 30 FPS。这样模型无法感知真实的时间节奏。

**Qwen2.5-VL 的改进：** 时间 ID 对齐真实秒数：

```
8 秒视频, FPS=2   → T-ID = [0, 0.5, 1.0, 1.5, ..., 7.5]   (间隔 0.5 秒)
8 秒视频, FPS=0.5 → T-ID = [0, 5, 10, 15]                   (间隔 5 秒)
```

RoPE 天然感知 ID 之间的距离 → 模型能理解"快速动作"和"慢速动作"的区别。

### 8.8 Qwen3-VL 的改进：Interleaved MRoPE（2025）

**问题：** 传统 MRoPE 把 T/H/W 各自分配一整块连续的维度，这意味着 T 信息全在低频通道，H/W 信息全在高频通道，频谱分配不均衡。

**改进：** 不按轴切块，而是在每个频率上交错分配 T/H/W：

```
传统 MRoPE:     [T₀ T₁ ... T₁₅ | H₁₆ ... H₃₉ | W₄₀ ... W₆₃]
Interleaved:    [T₀ H₁ W₂ | T₃ H₄ W₅ | T₆ H₇ W₈ | ...]
```

这样每个轴都能访问完整频谱，长视频的时间分辨率显著提升。

### 8.9 演进路线图

```
2021  RoPE (RoFormer)
      └─ 文本 1D 位置编码，LLM 标配
       ↓
2023  2D RoPE (ViT)
      └─ 图像 patch 的二维空间编码
       ↓
2024  MRoPE (Qwen2-VL)
      └─ T/H/W 三维统一 → 文本 图像 视频 一套编码搞定
       ↓
2024  MRoPE + 绝对时间 (Qwen2.5-VL)
      └─ 时间维度对齐真实秒数 → 理解事件节奏
       ↓
2025  Interleaved MRoPE (Qwen3-VL)
      └─ 交错分配 → 全频谱均衡 → 长视频更强
       ↓
学术前沿  Multi-Head RoPE / 可学习 RoPE 频率 / LieRE
```

### 8.10 核心趋势一句话

> 位置编码的演进方向是：**从加法变旋转 → 从 1D 变多维 → 从整块切分变交错全频谱 → 从固定频率变可学习**。每一步都在让模型更好地"理解空间"。

## 9. 深层理解：ViT token 融合全局信息后，位置编码还有意义吗？

**所属：** 深度学习基础 / 位置编码本质

### 9.1 核心困惑

> "ViT 经过多层 self-attention 后，每个 token 已经融合了其他 token 的信息，不再只代表自己原来那个 patch。那位置编码还有什么用？这个 token 的'位置'还能叫位置吗？"

这个问题触及了 Transformer 中"信息"和"位置"的根本关系。

### 9.2 关键结论（先给答案）

> **ViT 的 token 不是"局部特征"，也不是"全局特征"，而是："带有空间锚点的全局感知特征"。**

```
token ≠ 一个 patch
token ≠ 一个区域
token = 一个"位置视角下的全局理解"
```

### 9.3 两种含义要分开：信息 vs 坐标

每个 ViT token 最终同时携带两种东西：

| | 信息内容 (content) | 空间锚点 (position) |
|---|---|---|
| 来源 | 全图（self-attention 融合） | 最初 patch + 位置编码 |
| 含义 | "我知道什么" | "我从哪儿出发" |
| 会变吗 | 每层都在融合更多信息 | **不变**，始终写回同一个槽位 |

形式化理解：

```
最终 token_i ≈ 全局信息 + 一个空间坐标标签 (x_i, y_i)
```

### 9.4 为什么"锚点"不会丢？

Transformer 每一层做的操作是：

```
token_i ← Σ (attention_ij × token_j)
```

关键在于：**token_i 的更新始终写回 token_i 这个槽位。** 信息在流动，但位置索引不动。token 是"站在自己位置，看到了整个世界"。

### 9.5 类比：记者模型

每个 token 像一个记者：
- 站在固定地点（位置编码）
- 但可以通过通信知道全世界发生了什么（self-attention）

最后每个记者都知道全球新闻，但他报道的视角仍然是"我这个位置"。东京的记者 ≠ 巴黎的记者，因为**坐标不同**。

### 9.6 如果没有位置编码会怎样？

所有 token 变成"一堆全局语义向量，但谁也不知道谁的坐标在哪"：

```
token_A: 知道有猫，语义向量 ✓
token_B: 也知道有猫，语义向量 ✓
→ 但没人知道猫在哪 ✗
```

本质问题：

```
attention(q_i, k_j) = 纯语义相似度
```

模型无法区分"两个东西很像"和"两个东西很近"。

### 9.7 有位置编码后的本质变化

位置编码把 attention 变成了"语义 × 空间"的函数：

```
attention_ij ≈ f(语义_i, 语义_j, 距离(i, j))
```

- 两个 token 语义相似但距离很远 → 权重被压低
- 两个 token 语义不相似但距离很近 → 权重也可能高
- 同一只猫的两个 patch → 高语义相似 + 高空间接近 → 高注意力
- 两只不同猫的 patch → 高语义相似 + 低空间接近 → 注意力被抑制

### 9.8 一个更严格的视角

ViT 本质上在学一个函数：

```
f: (图像, 位置 i) → 表征向量
```

即每个 token 在学：**"整个图像在我这个位置的解释是什么？"**

这是比 CNN 更强的一点——CNN 的卷积核只看局部，而 ViT 每个 token 是"从自己的位置，看见整张图"。

### 9.9 对下游任务的意义

| 任务 | 怎么做 |
|---|---|
| "有没有猫？" | 看所有 token 的语义 |
| "猫在哪？" | 找语义含"猫"的 token，读它的位置锚点 |
| "左上角有什么？" | 找锚点在左上角的 token，读它的语义 |

如果没有位置 → 只能回答第一个问题，后两个统统答不了。

### 9.10 延伸：CNN 为什么不需要显式位置编码，而 ViT 必须？

**CNN：** 卷积核按固定空间模式滑动，权重共享。位置信息由**操作的顺序和感受野的层级**天然携带——靠近的两个像素在浅层就被同一个卷积核覆盖，远距离关系通过多层堆叠的 receptive field 逐步建立。位置是**操作结构**隐含的，不需要显式编码。

**ViT：** 第一层 self-attention 就做全局交互——所有 patch 两两之间直接算 attention，没有"先近后远"的层级约束。如果没有位置编码，模型根本不知道 patch A 在 patch B 左边还是右边。位置必须作为**输入数据**显式告诉模型。

> CNN 的位置是"结构赋予的"，ViT 的位置是"数据注入的"。

## 10. Attention 同时关注"位置"和"内容"

**所属：** 注意力机制本质

### 10.1 纠正一个之前的简化

之前讲"没有位置编码时 attention 就是纯语义相似度"，这是一种简化。更准确的说法是：

> **Attention 天然同时受位置和内容的影响，两者的区分程度取决于位置编码的方式。**

### 10.2 即使是最简单的 `token = content + pos`，Q 和 K 也能分离二者

很多人（包括之前的笔记）容易把"加法融合"理解成：位置和内容搅在一起，再也分不开了。但实际上：

```python
token_i = content_i + pos_i   # 加法融合

# 进入 attention：
q_i = W_Q @ token_i = W_Q @ content_i + W_Q @ pos_i   # Q 投影把两者重新混合
k_j = W_K @ token_j = W_K @ content_j + W_K @ pos_j   # K 投影也重新混合

# attention score 展开：
q_i · k_j = (W_Q @ content_i + W_Q @ pos_i) · (W_K @ content_j + W_K @ pos_j)

          = (W_Q @ content_i)·(W_K @ content_j)    ← 内容-内容
          + (W_Q @ content_i)·(W_K @ pos_j)        ← 内容-位置
          + (W_Q @ pos_i)·(W_K @ content_j)        ← 位置-内容
          + (W_Q @ pos_i)·(W_K @ pos_j)            ← 位置-位置
```

四项各司其职：

| 项 | 含义 | 例子 |
|---|---|---|
| content · content | 语义匹配 | "这两个 patch 都像猫耳朵" |
| content · pos | query 的内容找 key 的位置 | "我想找猫耳朵特征，它在哪个位置？" |
| pos · content | query 的位置找 key 的内容 | "我这个位置附近有什么特征？" |
| pos · pos | 纯空间偏置 | "不管内容是什么，近的 patch 多关注" |

**关键：** `W_Q` 和 `W_K` 是**可学习的**。训练过程中，模型可以学会让 Q 投影偏向"问位置"，K 投影偏向"答内容"，或者反过来。加法融合并没有堵死分离的路。

### 10.3 那加法融合和 MRoPE 的真正区别在哪？

不是"能不能区分位置和内容"——两者都能。区别在于**位置作用的"代数形式"**：

| | 加法融合（absolute / learned PE） | MRoPE（rotary PE） |
|---|---|---|
| 位置作用方式 | 加法：`token = content + pos` | 旋转：`token_rotated = rotate(token, pos)` |
| attention 中的位置效应 | Q·K 展开成 4 项**加法**组合 | Q·K 天然是相对位置的**乘法**函数 |
| 位置信息如何传递 | 通过 embedding 相加，每层都可能被 MLP 稀释 | 每层 attention 前直接旋转 Q、K，不经过 MLP |
| 对相对位置的归纳偏置 | 弱——需要数据学会"减法抵消绝对位置" | 强——数学上天然只依赖相对位置 |
| 长度外推 | 差——训练时没见过的位置 ID 对应未训练的 embedding 行 | 好——只依赖相对距离，与绝对 ID 范围无关 |

### 10.4 用公式看清本质区别

**加法位置编码的 attention score：**
```
score(i,j) = (W_Q·(c_i + p_i)) · (W_K·(c_j + p_j))
```
这是一种**任意函数**——W_Q 和 W_K 可以学会让它变成任何形式，包括相对位置。但需要**大量数据**来学到这种模式。

**RoPE 的 attention score：**
```
score(i,j) = (R_i·W_Q·c_i) · (R_j·W_K·c_j)
           = c_i^T · W_Q^T · R_{j-i} · W_K · c_j
```
`R_{j-i}` 天然只依赖相对位置 `j-i`，没有绝对坐标。这是**数学上保证的**，不需要数据来学。

### 10.5 一句话总结

> **加法融合没有"禁止"分离位置和内容（W_Q、W_K 可以学出来），但 RoPE/MRoPE 把"位置只取决于相对距离"写进了数学公式本身，给了模型一个更强的、不需要数据来学的归纳偏置。**

这也是为什么现在几乎所有模型都用 RoPE 而不是可学习的绝对位置编码——不是加减法的问题，是**加法需要数据来发现规律，旋转直接把规律写进了运算**。

## 11. 任务子空间：模型如何用高维向量的"方向"表达不同任务

**所属：** 表征学习 / 线性表示假说

### 11.1 核心直觉

> **Transformer 不是把不同任务分配到不同维度，而是把不同任务编码成 hidden state 空间中的不同"方向"。** 做任务 = 把 hidden state 投影到对应方向上，读出分量。

### 11.2 1D 投影：一个任务 = 一个方向

最简单的情形：hidden state `h ∈ R^4096`，某个任务由读出向量 `w ∈ R^4096` 定义：

```
score = w · h    ← 把 h 投影到 w 方向上，得到一个标量
```

**例子 —— lm_head 就是最典型的 1D 多方向读出：**

```
logits = W_lm · h    W_lm 形状: [152064, 4096]

每一行就是一个"方向"：
  W_lm["猫"] · h  → "输出'猫'的置信度"
  W_lm["狗"] · h  → "输出'狗'的置信度"
  W_lm["100"] · h → "输出坐标值'100'的置信度"
```

本质：**同时投影到 152064 个 1D 子空间上**。不是先降维再做任务，而是"任务本身定义了投影方向"。

### 11.3 多维子空间：复杂任务需要不止一个方向

有些任务不是一个方向能描述的，需要一个**低维子空间**（k 维）：

| 任务类型 | 为什么需要多维 | 子空间例子 |
|---|---|---|
| 2D 坐标 (x, y) | 两个独立自由度 | span{v_x, v_y} |
| bounding box | (x1, y1, x2, y2) 4 自由度 | 4 维子空间 |
| 句法结构 | 多层嵌套关系 | 几十维的子空间 |
| 物体类别 | 上千种物体需要区分 | 高维但远小于 4096 |

**数学表达：**

```
H_task = span{v₁, v₂, ..., v_k}       ← 任务子空间
h_task = P_task · h                   ← 投影到子空间
```

### 11.4 真实研究发现：语义信息确实存在低维子空间里

2025 年一项研究（Saglam et al., "Large Language Models Encode Semantics in Low-Dimensional Linear Subspaces"）分析了 11 个模型：

- 高层语义信息**稳定地存在于低维、线性可分的子空间**中
- 链式推理（CoT）的语义可以**用一个方向向量**捕捉
- 层越深，语义子空间的可分离性越强
- 可以通过在子空间做干预来**控制模型行为**（如检测恶意 prompt）

另一项关于数值推理的研究（"The Geometry of Numerical Reasoning", 2024）发现：
- LLM 在低维子空间中编码数值属性（如年份、大小）
- **因果干预**这些子空间会直接改变模型的比较结果（如 "C 罗出生早于梅西吗？"）

### 11.5 什么叫"任务接近"？—— 几何定义

两个任务接近 = 它们对应的方向/子空间在几何上接近。

**衡量方式：**

```
cosθ = (w₁ · w₂) / (|w₁| · |w₂|)
```

| cosθ | 含义 | 例子 |
|---|---|---|
| ≈ 1 | 高度接近 | "猫" vs "猫咪" |
| ≈ 0 | 无关 | "猫" vs "坐标x" |
| < 0 | 冲突/对立 | "正面" vs "负面" |

**具体例子：**

```
w_cat · w_dog   → 高余弦相似度（都描述宠物，共享很多语义特征）
w_cat · w_coord → 接近 0（语义 vs 空间，几乎正交）
w_left · w_right → 可能负相关（相反方向）
```

### 11.6 什么是"维度冲突"？

当 hidden state `h` 同时需要支持多个任务时，问题就来了：

```
h 同时需要：
  - 在 w_cat 方向上有强分量 → 判断"是不是猫"
  - 在 w_x 方向上有强分量  → 输出 x 坐标
  - 在 w_y 方向上有强分量  → 输出 y 坐标
```

如果这些方向不完全正交（即 `w_cat · w_x ≠ 0`），投影时会互相干扰：

```
读出 "猫"  →  w_cat · h
如果 w_cat 和 w_x 不正交，h 中的位置信息会泄露到语义读出里
```

**这就是为什么增大 hidden_size（如 4096→8192）有效：**

> R^d 里最多容纳 d 个互相正交的方向。d 越大，越容易让语义、位置、结构等不同任务族各自占据近似正交的子空间，减少互相干扰。

### 11.7 Superposition（叠加）：一个更激进的现象

2024 年的研究发现，神经网络实际上会做一件更极端的事——**Superposition（特征叠加）**：

> 模型可以在 d 维空间里表示**超过 d 个**特征，方法是用"几乎正交但略微重叠"的方向。

就像 3D 空间里可以塞下超过 3 个"近似正交"的向量（虽然数学上不可能完全正交，但实际中角度 > 80° 就够用了）。

这也解释了为什么单个神经元常常是**多语义的（polysemantic）**——它同时参与了多个特征的表示。

**但 Superposition 也有代价：** 当太多特征挤在同一空间时，会发生"特征吸收"（Feature Absorption）——一个更宽泛的特征被更具体的特征"吃掉"。比如 "starts with S" 的神经元在遇到 "short" 时不再激活，因为 "short" 这个具体词的特征把宽泛特征覆盖了。

### 11.8 线性表示假说的两种版本

| 弱版本 (Weak LRH) | 强版本 (Strong LRH) |
|---|---|
| **部分**特征是线性方向 | **所有**特征都是线性方向 |
| 广泛被实验支持 | 存在明确反例 |
| 线性探针、steering vector 等方法的基础 | 如 RNN 学习用"数量级"而非"方向"表示位置 |

**当前共识：** 弱 LRH 基本成立，尤其是在 Transformer 的深层。但强 LRH 不成立——存在非线性表示（如环形特征表示星期/月份，基于幅度的特征编码等）。

### 11.9 回到 VLM 的具体场景

现在你可以把 VLM 的 hidden state 理解成：

```
h ∈ R^4096

h = 语义分量(大)  +  空间分量(中)  +  句法分量(小)  +  ...
     ↑                  ↑                ↑
  w_cat 方向附近    w_x, w_y 子空间    其他方向
                   (≈2维)
```

```
lm_head 做的事：
  logits["猫"]  = W["猫"] · h   ← 主要读语义分量
  logits["100"] = W["100"] · h  ← 主要读空间分量（x坐标值）
  logits["200"] = W["200"] · h  ← 主要读空间分量（y坐标值）
```

**关键：** 同一个 h，不同的 W 行读出不同的东西。W 在训练中自然分化——语义类 token 的行学会对语义分量敏感，数字类 token 的行学会对空间分量敏感。

### 11.10 训练如何让任务方向变得"可分"？—— 梯度冲突驱动解耦

**核心结论：**

> **模型不会"主动规划子空间"，而是通过梯度下降，让不同任务的读出方向在梯度冲突的推动下，逐渐旋转到更可分（更接近正交）的方向上。** "任务解耦"不是显式设计出来的，而是梯度冲突被最小化的副产品。

#### 11.10.1 冲突的数学来源

假设两个任务 A（语义）和 B（位置），分别由读出向量 `w_A` 和 `w_B` 定义：

```
y_A = w_A · h
y_B = w_B · h
```

总 loss：`L = L_A + L_B`

对 hidden state `h` 的梯度：

```
∂L/∂h = (∂L_A/∂y_A) · w_A  +  (∂L_B/∂y_B) · w_B
```

**关键：`h` 的更新方向 = 两个任务方向的线性组合。** 如果 `w_A` 和 `w_B` 方向很接近，两个任务的梯度就高度耦合，互相牵制。

#### 11.10.2 冲突的典型场景

```
任务 A（语义）：想让 h 在 w_A 方向上的分量 ↑
任务 B（位置）：想让 h 在 w_A 方向上的分量 ↓（因为 w_B 和 w_A 方向接近）

→ 梯度互相抵消 → loss 降不下去 → 模型被迫"想办法"
```

#### 11.10.3 模型怎么解决？两条路

**方法 1：调整表示 `h`（分解信息）**

让 `h` 内部自然分化：

```
h = h_A + h_B
  h_A 主要影响 w_A 的读出
  h_B 主要影响 w_B 的读出
```

**方法 2：调整读出方向 `W`（旋转分离）**

让 `w_A` 和 `w_B` 往更正交的方向旋转：

```
训练前: w_A ⟂̸ w_B  （不正交，干扰大）
训练后: w_A ⟂ w_B   （近似正交，干扰小）
```

#### 11.10.4 "旋转"到底是谁在转？

不是单独 `h` 在转，也不是单独 `W` 在转——**整个表示空间在被共同重参数化**：

```
h ← f(x; θ)      ← Transformer 的每一层都在不断旋转+拉伸空间
W ← 可学习参数   ← lm_head 的行也在被梯度推着走
```

每次梯度更新，前向网络里的线性变换矩阵 `W_Q, W_K, W_V, W_O, W_mlp` 都在做微小的旋转和拉伸，多层叠加的效果就是：**逐步把纠缠的表示拉开。**

#### 11.10.5 与 PCA/ICA 的类比

模型在做的事类似于 PCA（找主方向）和 ICA（找独立成分），但有一个本质区别：

| | PCA / ICA | 神经网络 |
|---|---|---|
| 驱动因素 | 统计方差 / 独立性 | **任务 loss 的梯度** |
| 方向来源 | 数据分布决定 | 任务需求决定 |
| 优化目标 | 重构误差最小 | 下游任务误差最小 |

> **这里的"方向"不是数据方差最大的方向，而是任务 loss 逼出来的方向。**

#### 11.10.6 为什么 Transformer 特别擅长这件事？

因为每一层都是：

```
h ← W_2 · σ(W_1 · h)
     ↑        ↑
  线性旋转  非线性重分布
```

多层堆叠后，拥有了任意逼近"空间重排"的能力——可以把纠缠的表示逐步拆开。

#### 11.10.7 回到工程解法，用这个视角重新理解

| 工程做法 | 在这个视角下的本质 |
|---|---|
| 增大 hidden_size | 提供更多可正交的方向，减少维度竞争 |
| 特殊 token（如 `<|box_start|>`） | 不同任务激活不同子空间，减少梯度冲突 |
| 分阶段训练（先语义后定位） | 先固定一部分方向，再在剩余空间里种新任务 |
| 多 projector（主投影器 + 辅助投影器） | 在输入端就做预解耦，减少后续层的解耦压力 |

#### 11.10.8 一个精确的类比

想象调音台：

- 两个乐器（任务）声音混在一起
- 调音台（网络）可以旋转旋钮（线性变换）

如果两个声音互相干扰 → 你把它们调到不同声道，或不同频率 → 这就等价于：**把信号旋转到不同子空间。**

#### 11.10.9 为什么有些任务永远无法完全正交？

**核心结论：** 语义与位置这类任务无法完全正交，不是因为模型能力不够，而是因为**它们在数据分布和任务定义上本来就存在统计依赖与功能耦合**。强行完全正交反而会损失有用信息、增加优化难度。

#### 11.10.10 精确澄清：不是"维度开关"，而是"方向强弱"

一个非常容易踩的误解：

> ❌ 某些维度完全不动，loss 只更新部分维度
> ✅ 所有维度都在动，但梯度在某些方向上"投影更大"，在另一些方向上"接近为 0"

```
不是 维度级别的开/关
而是 方向级别的强/弱
```

##### 用线性代数精确写出来

hidden 空间确实会"近似分解"：

```
h = h_semantic + h_position + h_shared
```

但注意：这不是"维度切块"，而是"方向分解"。更准确是：

```
h ∈ R^d

存在一些方向：
- 子空间 S_semantic
- 子空间 S_position
- 子空间 S_shared
```

这些子空间：
- **近似正交（不是严格）**
- **可以是旋转后的任意方向（不是固定维度块）**

##### 梯度是"投影到这些方向上"的

梯度：

```
∂L/∂h = g_A · w_A + g_B · w_B
```

假设：
- `w_A` 主要落在语义子空间
- `w_B` 主要落在位置子空间

那么梯度自然会：

```
在 S_semantic 上很大
在 S_position 上很大
在其他方向上很小
```

> **梯度自动在"任务相关子空间"里流动，在无关子空间里几乎不流动。**

##### 几何直觉：3D 空间类比

想象一个 3D 空间：

- x 轴：语义
- y 轴：位置
- z 轴：无关噪声

| 任务 | 梯度方向 |
|---|---|
| 语义 | 沿 x 方向 |
| 位置 | 沿 y 方向 |
| z 轴 | 梯度 ≈ 0 → 几乎不更新 |

> **不是"z 轴被冻结了"，而是"梯度在 z 方向上自然消失"。**

##### 核心公式（更严格版）

```
∂L/∂h = Σ_i g_i · w_i
```

如果某个方向 `v` 满足：

```
v ⟂ w_A
v ⟂ w_B
```

那么：

```
v · (∂L/∂h) ≈ 0
```

→ **这个方向几乎不会被更新。**

##### 常见误区

很多人会误以为：

```
[前 512 维 = 语义]
[后 512 维 = 位置]
```

❌ 这是错的。真实情况是：

```
语义 = 某个旋转方向（可能是所有维度的线性组合）
位置 = 另一个旋转方向
```

##### 更深一层理解

> **模型学到的不是"哪个维度属于哪个任务"，而是：哪些方向对哪些任务有用。**

这意味着：
- 子空间是**旋转的**
- 不是固定坐标轴
- 每一层都会重新混合

##### 一句话总结

> **不是"loss 只更新某些维度"，而是"梯度只在与任务相关的方向上有分量"，因此这些方向对应的表示被更新得多，而其他方向几乎不动。**

这层理解的工程延伸：

- **为什么 LoRA / adapter 有效？** → 本质是在"低秩地插入新子空间"，不干扰已有方向
- **为什么 multi-head attention 有效？** → 本质是多子空间并行读出，每个 head 关注不同方向

#### 11.10.11 升级版理解：梯度冲突驱动解耦的完整逻辑链

本节是对 11.10 核心结论的"校正 + 补全"，让表述更严谨。

##### 第一步：承认原始直觉是对的，但加一个前提

> **模型不会"主动规划子空间"，而是通过梯度冲突，让任务方向逐渐变得可分（接近正交）**

这是一个非常好的近似理解，但要注意：

- 模型并不是"为了正交而正交"
- 它真正优化的是：**总 loss 最小化**
- "正交 / 可分"只是**在某些条件下的最优解形态之一**

##### 第二步：为什么"梯度不分开"，却不会彻底乱？

梯度确实是直接叠加的，没有隔离机制：

```
∂L/∂h = ∂L_A/∂h + ∂L_B/∂h
```

那为什么没崩？

**梯度冲突的定义：**

```
g_A = ∂L_A/∂h,  g_B = ∂L_B/∂h
g_A · g_B < 0  → 梯度冲突
```

冲突的结果是更新方向互相抵消，两个任务都学不好，loss 卡住。

**模型的"自救机制"：**

当冲突持续存在时，梯度下降会推动模型改变表示 `h`，使得：

```
g_A · g_B ↑ （变得不那么负）
```

一个自然的方式就是让任务依赖不同的表示方向：

```
w_A ⟂ w_B（近似正交）
```

于是：

```
g_A ≈ 只作用在子空间 A
g_B ≈ 只作用在子空间 B
```

✔ 这一步就是"隐式解耦"。

##### 第三步：精确化表述——不是"读出方向正交"，而是"有效梯度子空间分离"

之前写的 `w_A ⟂ w_B` 可以进一步精确为：

> **梯度在表示空间中的投影逐渐分离**

即：

```
span(g_A) ⟂ span(g_B)
```

不一定是参数向量正交，而是：**任务对表示的"使用方式"变得不同。**

##### 第四步：为什么这种现象会"自然出现"？

因为**冲突是有损的**：

- 如果不解耦 → 更新效率 ↓，loss 降不下去
- 一旦解耦 → 每一步更新都更有效

所以在优化过程中：

> **"减少梯度冲突" = "更快下降 loss"**——这就是驱动力。

##### 第五步：关键限制——模型容量必须足够

这个结论只在**模型容量足够大**时成立。如果模型容量不够：

- 它**没法分出子空间**
- 任务 A / B 争夺同一组特征
- 出现**负迁移（negative transfer）**
- loss 互相拖累

##### 第六步：什么时候不会变正交？

| 情况 | 为什么 |
|---|---|
| 任务本来就相关（语法+翻译、检测+分割） | 最优解是**部分共享 + 部分分离**，不是完全正交 |
| 梯度本来一致（`g_A · g_B > 0`） | 模型会**强化共享表示**，而不是解耦 |
| 优化器/训练策略限制（强正则化、参数共享约束） | 可能导致**解耦不完全** |

##### 最终：升级版结论

原版本：
> 梯度冲突推动任务方向变得正交

升级版：

> 在多任务训练中，由于梯度是线性叠加的，若不同任务之间存在持续的梯度冲突（负内积），优化过程会推动模型调整表示，使各任务的有效梯度在表示空间中的投影逐渐分离，从而减少冲突。这种"任务解耦"并非显式设计，而是最小化总 loss 的过程中自然出现的副产物。

**一句话：** 不是模型"想解耦"，而是**不解耦就学不好，所以它被迫解耦。**

##### 延伸思考

这个框架可以继续解释：
- **为什么 Transformer 里会出现 head specialization（头专门化）？** → 不同 head 自然演化出不同子空间偏好
- **LoRA / 低秩适配为什么有效？** → 本质是在原空间里插入新的低秩子空间，不与已有方向冲突
- **为什么分阶段训练有效？** → 先固定一部分方向（预训练），再在剩余空间里种新任务（微调）

#### 11.11 无法完全正交的五层原因

##### 第一层：数据本身就不独立（统计耦合）

现实世界里：
- "猫"更常出现在地面、沙发（特定位置分布）
- "天空"通常在上方
- "道路"通常在下方

```
P(语义 | 位置) ≠ P(语义)
```

如果数据里本来就有关联，模型把它们完全分开反而更难优化——因为语义预测本来就可以利用位置先验，强行正交等于把有用信号切断。

##### 第二层：任务目标本身是"联合函数"

很多任务不是 `f(语义) + g(位置)`，而是 `f(语义, 位置)`。例如"左上角有没有猫？"：

```
这个任务本质 = 是否存在 (语义=猫 AND 位置=左上)
→ 这是语义 × 位置的交互项
→ 正交意味着没有交互 → 任务需要交互 → 不可能完全正交
```

##### 第三层：表示容量有限（维度不够）

```
hidden_size = d
语义需要 k₁ 维，位置需要 k₂ 维，交互还需要 k₃ 维
如果 k₁ + k₂ + k₃ > d → 只能压缩、重叠、共享方向
→ 语义子空间 ∩ 位置子空间 ≠ ∅
```

##### 第四层：非线性网络会"重新混合"

即使某一层做到了 `语义 ⟂ 位置`，下一层：
```
h' = W₂ · σ(W₁ · h)
```
线性混合 + 非线性重组 → 正交又被打破了。**正交不是稳定不变的结构，而是不断被打破再重组。**

##### 第五层：优化目标不鼓励正交

训练目标是 `minimize loss`，不是 `minimize loss + 保持子空间正交`。模型会选择最易降低 loss 的表示，而不是最干净分离的表示。

##### 一个直觉例子

任务 A：识别"人"；任务 B：预测"人在哪"。

如果完全正交：
- A 完全不知道位置 → 不知道关注哪里
- B 完全不知道人的概念 → 不知道找什么
- → 两个任务都做不好

**正确做法是：部分共享 + 部分独立。**

#### 11.11.1 真实结构：共享 + 专属

不是：

```
语义 ⟂ 位置
```

而是：

```
        共享子空间 (H_shared)
       /                    \
  语义专属 (H_semantic)   位置专属 (H_position)

且 H_shared ≠ ∅   ← 这才是正常的
```

#### 11.11.2 最终总结

> **模型不是在追求"把任务分开"，而是在追求"用最少的维度表达最有用的信息"。如果任务之间本来就相关，它们就必然共享方向，无法完全正交。这种"部分共享 + 部分独立"的结构，是表示学习中解耦与表达力之间的最优折中。**

**一句话总结整个 11 章：**

> **模型不是把任务分配进不同的"格子"（维度），而是把任务写成不同的"箭头"（方向）。训练过程 = 梯度冲突不断出现 → 模型调整表示和读出方向 → 让不同任务在几何上更可分 → 等价于把信息"旋转"到不同子空间。但不是所有任务都能完全正交——真实结构中存在共享子空间，因为任务之间的统计依赖和功能交互天然要求部分重叠。**

### 11.12 多任务 loss 不会互相干扰吗？—— 梯度分流机制

**问题：** loss 是多任务加起来的：

```
L = L_A + L_B + L_C + ...
```

梯度直接叠加：

```
∂L/∂h = ∂L_A/∂h + ∂L_B/∂h + ∂L_C/∂h
```

没有任何显式隔离机制——**那为什么不会彻底乱掉？**

**答案：模型会自己调整表示，让梯度"尽量不冲突"。**

##### 冲突 → 自救的完整链条

**如果两个任务方向冲突（w_A ≈ w_B）：**

```
g_A·w_A + g_B·w_B  ≈  互相抵消
```

→ loss 降不下去
→ 模型被迫改变结构
→ 逐渐变成 w_A ⟂ w_B

**于是：**

```
∂L_A → 只影响方向 A
∂L_B → 只影响方向 B
```

##### 关键结论

> **多任务 loss 没有显式分离，但表示空间会被训练成"隐式分离"。**

换句话说：

```
不是 loss 被拆开
而是 梯度在空间中被分流
```

### 11.13 LoRA / Adapter = 低秩插入新子空间

##### 标准全量微调

```
W ← W + ΔW
ΔW ∈ R^{d×d}   （满秩）
```

→ 可以改变整个空间结构，代价大，容易破坏原能力。

##### LoRA 的做法

```
ΔW = A · B

A ∈ R^{d×r}
B ∈ R^{r×d}
r << d         ← 低秩
```

##### 几何拆解

```
ΔW h = A (B h)
```

| 步骤 | 操作 | 含义 |
|---|---|---|
| 第一步 | `z = B h ∈ R^r` | 把原始表示**压缩到 r 维子空间** |
| 第二步 | `A z ∈ R^d` | 再映射回原空间 |

##### 本质

> **LoRA = 在原空间中"插入一个 r 维的新通道/子空间"**

```
原模型：只能用已有方向表达任务
LoRA：新增 r 个方向专门服务新任务
```

这就是**给新任务一个"独立活动空间"，减少它和旧任务抢方向。**

##### 为什么"低秩"是关键？

```
rank(ΔW) ≤ r
```

→ **你只允许模型改动"少数几个方向"**

这带来两个效果：
1. 不破坏原有子空间结构
2. 新任务被限制在"局部子空间"里学习

##### 一句话总结

> **LoRA 不是在改整个模型，而是在原空间里"外挂一个低维子空间"给新任务用。**

### 11.14 Multi-Head Attention = 多子空间并行

##### 单头 attention

```
Q = W_Q h
K = W_K h
V = W_V h
```

→ 所有信息在**一个投影空间里混**。

##### Multi-head 在干嘛？

```
head_i:
    Q_i = W_Q^i h
    K_i = W_K^i h
    V_i = W_V^i h

concat(head_1, head_2, ..., head_n) → W_O
```

每个 head 的 `W_Q^i, W_K^i, W_V^i` 都是**不同的线性投影**。

→ 每个 head = 一个不同的"观察子空间"。

##### 直观理解

| head | 可能专注的子空间 |
|---|---|
| head 1 | 语法关系 |
| head 2 | 长距离依赖 |
| head 3 | 位置结构 |
| head 4 | 语义相似性 |

##### 为什么这就是"多子空间并行"？

```
h → 多个子空间 → 各自计算 attention → 再融合
```

##### 与梯度解耦的关系

如果只有一个 head：

```
所有任务共享一个空间 → 容易冲突
```

多头：

```
不同任务可以"占不同 head"
→ 天然减少梯度冲突
```

##### 更本质的说法

> **Multi-head attention = 强制模型在多个子空间中"并行建模不同关系"。**

### 11.15 统一框架：子空间动力学

把三件事串起来：

```
多任务 loss
  → 多个梯度叠加 → 产生冲突
  
Transformer 的解决方案
  → 通过训练 → 旋转空间 → 分离方向

工程增强手段
  → LoRA：给新任务额外子空间
  → Multi-head：一开始就提供多个子空间
```

##### 最终统一理解

> **现代神经网络的本质 = 在高维空间中不断创建、旋转、分配"子空间"，让不同任务的梯度可以各走各的路，同时在需要时共享。**

##### 延伸方向（同一框架下可继续理解的问题）

| 现象 | 子空间视角的解释 |
|---|---|
| attention head 退化（某些 head 没用） | 该 head 的子空间没人用，被其他 head 覆盖了 |
| MoE（Mixture of Experts） | 离散子空间分配：用 router 决定激活哪些专家（子空间） |
| representation collapse | 子空间坍缩：所有方向挤到少数几个，失去表达能力 |

## 13. Attention 权重是标量：如何与 V 相乘？

**所属：** 注意力机制基础

### 13.1 核心机制

**注意力分数是一个标量（单个值），这个标量会乘以 V 中该 token 的所有维度，然后逐元素相加。**

### 13.2 具体数值示例

假设 3 个 token，每个 token 的 V 是 4 维向量：

```
V 矩阵 (3个token × 4维)：
Token1: [1, 2, 3, 4]
Token2: [5, 6, 7, 8]
Token3: [9, 10, 11, 12]
```

Token1（作为查询）的注意力权重：
```
a1 = [0.7, 0.2, 0.1]   # 对Token1关注0.7，Token2关注0.2，Token3关注0.1
```

**计算 Token1 的输出：**

```
输出 = 0.7 × [1, 2, 3, 4] + 0.2 × [5, 6, 7, 8] + 0.1 × [9, 10, 11, 12]
    = [0.7, 1.4, 2.1, 2.8] + [1.0, 1.2, 1.4, 1.6] + [0.9, 1.0, 1.1, 1.2]
    = [2.6, 3.6, 4.6, 5.6]
```

### 13.3 两种等价的理解角度

**角度1：逐向量处理** — 先给每个 V 向量乘以一个标量，再相加：

```
输出 = 0.7×[1,2,3,4] + 0.2×[5,6,7,8] + 0.1×[9,10,11,12]
    = [2.6, 3.6, 4.6, 5.6]
```

**角度2：按维度分别处理** — 对于每个维度，把该维度上所有 token 的值乘以对应权重后相加：

```
维度1: 0.7×1 + 0.2×5 + 0.1×9 = 2.6
维度2: 0.7×2 + 0.2×6 + 0.1×10 = 3.6
维度3: 0.7×3 + 0.2×7 + 0.1×11 = 4.6
维度4: 0.7×4 + 0.2×8 + 0.1×12 = 5.6
```

两种方式完全等价。

### 13.4 矩阵运算走的是"维度级"路线

代码中：

```python
# attention_weights: (batch, n, n)  第i行是第i个token的注意力分布
# V: (batch, n, d_model)

output = attention_weights @ V  # (batch, n, n) × (batch, n, d_model)
                                # = (batch, n, d_model)
```

矩阵乘法在底层同时对所有维度计算，效率远超逐个维度循环。

### 13.5 关键观察

1. **注意力是一个标量**：`0.7, 0.2, 0.1` 这些都是单个数字
2. **同一权重应用于所有维度**：Token1 的 4 个维度 `[1,2,3,4]` 都乘以 `0.7`
3. **逐元素相加**：在同一个维度位置上，把三个 token 的值加起来

### 13.6 为什么这样设计？

如果不同维度用不同权重，就变成了：在维度1上可能更关注 Token1，在维度2上可能更关注 Token2。这会破坏注意力的语义——**应该让模型决定"关注哪个 token"，而不是"关注 token 的哪些维度"。**

### 13.7 总结

| 理解 | 正确性 |
|---------|--------|
| 注意力是一个值（标量） | ✅ |
| 这个标量乘以V的所有维度 | ✅ |
| 所有token的结果在相同维度相加 | ✅ |

## 14. Attention 掩码机制：子句掩码、因果掩码、Padding 掩码

**所属：** 注意力机制 / Grounding DINO

### 14.1 Grounding DINO 的子句掩码（对齐掩码）

**特殊 token：** `[CLS]`, `[SEP]`, `.`, `?` 被当作子句分隔符。

**作用：** 以这些分隔符为边界，将文本切成独立块，不同块之间互相看不到。

**具体例子：** 输入 `"[CLS] a cat . a dog [SEP]"`

```
生成的注意力 mask（✓ = 可以关注）：
           [CLS] a  cat  .  a  dog [SEP]
    [CLS]    ✓
    a           ✓  ✓   ✓
    cat         ✓  ✓   ✓
    .           ✓  ✓   ✓
    a                       ✓  ✓   ✓
    dog                     ✓  ✓   ✓
    [SEP]                           ✓
           ├── 块1: "a cat" ──┤  ├── 块2: "a dog" ──┤
```

**关键特征：**
- 所有 token 拼接在一起**一次送入** BERT，不是拆开多次跑
- 块内 token 互相可见，块间隔离
- 每块内 position_id 独立从 0 重新编号，避免前面长句子导致后面位置编码偏移过大

**代码位置：** `bertwarper.py:224` → `generate_masks_with_special_tokens_and_transfer_map()`

```python
# groundingdino.py:132
self.specical_tokens = self.tokenizer.convert_tokens_to_ids(["[CLS]", "[SEP]", ".", "?"])
```

### 14.2 三种掩码对比

| | 因果掩码 | Padding 掩码 | 子句掩码（对齐掩码） |
|---|---|---|---|
| **目的** | 第 i 位不能看到 i+1 之后 | 不能让模型关注无效 token | 不同子句之间互相不能看到 |
| **形状** | 下三角矩阵 | 按 batch 中每个样本的有效长度 | 块对角矩阵 |
| **谁生成** | 固定规则 | tokenizer 的 `attention_mask` | 特殊 token 动态切分 |
| **用在哪种模型** | Decoder（GPT 类） | 所有 Transformer | BERT（双向编码器） |
| **效果** | 保证自回归，不能偷看未来 | 忽略 padding 位置的噪声 | 子句语义独立，位置编码局部化 |

### 14.3 掩码何时生效：softmax 之前，不是之后

**标准流程：**

```
attention_scores = Q @ K^T / sqrt(d)          # 原始分数，如 [0.5, -0.3, 2.1, ...]
attention_scores = attention_scores + mask_bias  # mask 位置加 -10000
attention_weights = softmax(attention_scores)    # softmax(-10000) ≈ 0
output = attention_weights @ V                   # 被 mask 的 token 贡献 ≈ 0
```

三种掩码都是这个机制——只是 mask 的形状不同，不是操作时机不同。

### 14.4 BERT 是纯编码器结构

```
BERT:  [Transformer Encoder] × 12 层  →  双向 self-attention，无 causal mask
GPT:   [Transformer Decoder] × 12 层  →  单向 causal self-attention
T5:    [Encoder] × N + [Decoder] × N  →  完整编码器-解码器
```

BERT 每个 token 能同时看到左右两边，所以上面那种块对角 mask 是有意义的（限制子句间交互但保留子句内双向）。如果是 GPT 的 causal decoder，本身就是下三角，token 天然只能向左看。

### 14.5 掩码不跳过计算：每个词依然完整计算 Q·K

mask 位不是"不算了"，而是"算出分数再强制归零"。

以输入 `"a cat . a dog"` 中 token "dog" 为例：

```
1. Q_dog · K 全部算一遍：
   scores = [0.5,  0.3, -0.1, -0.2,  0.8,  0.6, -0.4]
            [CLS]  a   cat    .     a    dog  [SEP]
              ↑                         ↑
         块间仍然算了，分数正常        同块分数也正常

2. 加 mask_bias（非块内位置 → -∞）：
   scores = [-∞,  0.3, -0.1, -0.2,  0.8,  0.6,  -∞]
            [CLS]  a   cat    .     a    dog  [SEP]
              ↑                             ↑
         前一块的 token                后一块末尾 token
         被压成 -∞                     被压成 -∞

3. softmax 后：
   weights = [0,  0.12, 0.08, 0.07, 0.41, 0.32,  0]
             ↑                                   ↑
         权重为 0，不参与 V 加权求和              为 0

4. 最终输出：
   output = 0.12·V_a + 0.08·V_cat + 0.07·V_. + 0.41·V_a_block2 + 0.32·V_dog
```

**核心：不是"不计算 Q·K"，而是"算了但 softmax 后权重趋近于 0"。** 因果掩码也是同一机制——未来 token 的 Q·K 照算，但被压成 -∞ 后权重为 0。

## 15. 多头注意力：计算量与结果的平衡

**所属：** 注意力机制

### 15.1 核心结论

> 多头注意力确实把 n² 的计算做了 h 次，但不是重复计算，而是在 h 个不同表示子空间中并行建模不同关系，最后融合成 n 个输出。

### 15.2 单头 vs 多头

**单头注意力：**
```
Q, K, V ∈ R^(n × d)
→ attention matrix: n × n
→ output: n × d
```

**多头注意力（h=12 头）：**
```
Q → 投影 → 12 个 Q_i，每个 R^(n × d/12)
K → 投影 → 12 个 K_i，每个 R^(n × d/12)
V → 投影 → 12 个 V_i，每个 R^(n × d/12)
```

每个 head 做一次 n × n 注意力（共 h 次），但每个 head 在**不同的低维子空间**中计算。

### 15.3 为什么要降维（d/h 而不是 full-d）？

核心问题：**为什么不直接做 12 个 full-d 的 attention？**

**计算量视角：**

| 方案 | 每次 Q·K^T 复杂度 | 总复杂度 | 参数增量 |
|---|---|---|---|
| 单头 full-d | O(n²d) | O(n²d) | 3d² (Q,K,V投影) |
| 12 头 full-d（不降维） | O(n²d) × 12 | O(12 n²d) | 36d² |
| 12 头 d/12（标准做法） | O(n² · d/12) × 12 | **O(n²d)** | 3d² |

**关键：** 标准多头注意力的总计算量 = 单头 full-d 的计算量。降维让 h 个头拼起来恰好等于原来的 d 维。不降维的话计算量直接翻 h 倍，但表达能力不会翻 h 倍——因为 12 个 full-d 头有大量冗余，每个头的能力已经饱和。

**表达能力视角：**

```
12 个 full-d 头：
  每个头都能看到完整的 d 维空间 → 高度冗余 → 12 个头可能学到相似模式 → 浪费

12 个 d/12 头：
  每个头只看到一个低维切片 → 被迫专精 → 不同头自然分化出不同关系模式
  例如：
    head 1：关注主谓关系
    head 2：关注指代（he/she）
    head 3：关注位置
    head 4：关注语义相似
```

**降维 = 信息瓶颈 = 隐式正则化**。它强制每个头在有限的 d/12 维里"挤"出最有效的信息提取方式，而不是在 full-d 空间里舒舒服服地学成一堆高度相似的东西。

### 15.4 输出融合

```
每个 head 输出: (n × d/12)
拼接: concat → (n × d)
再过 W_O: (n × d) → (n × d)
```

不是得到 12n 个输出，而是融合回 n 个。

### 15.5 与子空间解耦的联系

这和之前第 11 章讨论的"梯度冲突 → 子空间分离"是同一思想的结构化版本：

> 多头注意力 = **人为提供多个子空间**，而不是等模型自己慢慢旋转出来。

不同 head 演化出不同子空间偏好（head specialization），天然减少了梯度冲突——不同关系类型在不同 head 里建模，它们的梯度在空间中被分流。

### 15.6 一句话总结

> 降维（d/h）让 h 个头的总计算量恰好等于 1 个 full-d 头，同时用信息瓶颈强制每个头学不同的关系模式——用相同的算力换来了更丰富的表示能力。

## 16. Backbone 输出的不是 token，是 2D 特征图

**所属：** 模型架构 / Grounding DINO

### 16.1 核心区别

| | ViT（如 DETR） | Swin/ResNet（如 Grounding DINO） |
|---|---|---|
| 输出格式 | token 序列 `[B, N, C]` | 2D 特征图 `[B, C, H, W]` |
| 空间结构 | 已展平，丢失 2D 关系 | 保留，H×W 明确 |
| 位置编码 | 额外加 position embedding | 有独立的位置编码分支 `poss` |
| 投影方式 | `nn.Linear(C, d)` | `nn.Conv2d(C, d, 1)` |

即使 Swin Transformer 内部做了 self-attention，它用 window-based attention 保持了 2D 空间结构，输出**仍是特征图**。

### 16.2 Grounding DINO 完整操作链路

```
Backbone (Swin/ResNet)
    │
    ▼  多尺度 2D 特征图，各层通道数不同
features[0]: [B, 512,  100, 150]   ← Layer3
features[1]: [B, 1024, 50,  75 ]   ← Layer4
features[2]: [B, 2048, 25,  38 ]   ← 额外下采样
features[3]: [B, 2048, 13,  19 ]   ← 再下采样
    │
    ▼  input_proj：每个特征图用各自的 1×1 Conv
input_proj[0](src) → [B, 256, 100, 150]
input_proj[1](src) → [B, 256, 50,  75 ]
input_proj[2](src) → [B, 256, 25,  38 ]
input_proj[3](src) → [B, 256, 13,  19 ]
    │
    ▼  flatten：展平成 token
src.flatten(2).transpose(1,2)
[B, 256, 100, 150] → [B, 15000, 256]
[B, 256, 50,  75 ] → [B, 3750,  256]
[B, 256, 25,  38 ] → [B, 950,   256]
[B, 256, 13,  19 ] → [B, 247,   256]
    │
    ▼  concat：拼成一个大序列
total: [B, 19797, 256]
    │
    ▼  送入 Transformer Encoder
```

### 16.3 为什么 Grounding DINO 需要保留 2D 特征图？

因为 deformable attention 需要知道每个特征点的**2D 空间坐标**来计算采样位置。如果用 ViT 那种已经展平的 token，空间信息就丢失了，后续的可变形注意力无法基于 2D 参考点做稀疏采样。

## 17. GroupNorm：为什么 input_proj 用 GroupNorm 而不是 BatchNorm/LayerNorm

**所属：** 模型架构 / 归一化

### 17.1 三种归一化对比

```
BatchNorm:           LayerNorm:              GroupNorm (groups=32):
每个通道在全 batch    每个样本所有通道          每组 8 个通道独立归一
上归一化             归一化

256 通道             256 通道                 32 组 × 8 通道
[===c1===]          [c1..c256 一起]          [c1..c8][c9..c16]...[c249..256]
[===c2===]                                    ↑       ↑           ↑
    ...                                       各自独立归一化
```

### 17.2 对比表

| | BatchNorm | LayerNorm | GroupNorm (groups=32) |
|---|---|---|---|
| 依赖 batch 大小 | ✅ 依赖 | ❌ 不依赖 | ❌ 不依赖 |
| 小 batch 稳定性 | ❌ batch 小则统计不准 | ✅ | ✅ |
| 适合检测任务 | 一般 | ❌ 检测里表现差 | ✅ |
| 通道间关系 | 完全独立 | 全部混合 | 组内共享统计量 |

### 17.3 为什么这里选 GroupNorm？

1. **检测任务 batch_size 很小**（通常 2~4），BatchNorm 的均值和方差估计不稳定
2. **LayerNorm 在 CNN 特征图上表现差**，它对所有通道做全局归一化，丢失了不同通道的独立性
3. **GroupNorm 是折中**：不依赖 batch，同时保留通道组内的相对关系

### 17.4 groups=32 的设定

`hidden_dim=256`，`groups=32` → 每组 256/32 = 8 个通道。

不大不小：组太少（如 groups=1 = LayerNorm）会丢失通道多样性，组太多（如 groups=256 = InstanceNorm）则完全没有通道间交互。32 是检测任务的经验最优值，来自 GroupNorm 原论文（Wu & He, ECCV 2018）。

## 18. ContrastiveEmbed：无参数的"分类头"

**所属：** 模型架构 / Grounding DINO

### 18.1 它做什么

`ContrastiveEmbed` 不是传统分类头，**没有可学习参数**。它只做一件事：

```python
res = x @ y.transpose(-1, -2)   # 纯矩阵乘法，无权重
# x: [B, N_queries, 256]  ← 视觉特征（object query）
# y: [B, L, 256]          ← 文本特征（Encoder 增强后的 token embedding）
# res: [B, N_queries, L]  ← 每个 query 与每个文本 token 的点积相似度
```

然后两步后处理：
1. **masked_fill**：将 padding token 的分数置为 -∞
2. **pad to max_text_len**：补到固定长度 256

### 18.2 与 YOLO 分类头的对比

| | YOLO 分类头 | Grounding DINO ContrastiveEmbed |
|---|---|---|
| **结构** | `nn.Linear(256, 80)` 有可学习权重 W | 无参数，纯点积 `x @ y^T` |
| **类别数** | 固定 80 类（COCO），训练时就定死 | 动态，由输入文本决定 |
| **"猫"的分数来源** | `W["猫"] · x` — 权重矩阵第"猫"行的固定向量 | `x · y_cat` — 当前输入中"cat"的实时文本向量 |
| **新增类别** | 改头、重新训练 | 换个 prompt 文本就行，零样本 |
| **推理输出** | 80 个分数 | L 个分数，对应输入文本的每个 token |

### 18.3 本质区别

```
YOLO（闭集）：
  x [256] → Linear(256, 80) → [80] → argmax → "cat"
            ↑
        W 固定，80 个类别各一行，终生不变

Grounding DINO（开集）：
  x [256] · y_text^T → [L] → "cat" 是其中某个 token 的分数
  ↑         ↑
  视觉      文本 token 向量，输入变它就变
```

**一句话：** YOLO 的类别空间由 `nn.Linear` 的权重矩阵固化，Grounding DINO 的类别空间由当前输入的文本动态定义。这就是开集检测的根本——不是"从固定类里选"，而是"和当前描述文本里的词比相似度"。

## 19. 两阶段模式：Encoder 如何生成初始候选框

**所属：** 模型架构 / Grounding DINO

### 19.1 两种模式

| | 单阶段 (`"no"`) | 两阶段 (`"standard"`) |
|---|---|---|
| 初始 query 来源 | 可学习 Embedding（随机初始化） | Encoder 输出中选 Top-K |
| 初始参考点 | 随机初始化 | Encoder 预测的高置信度位置 |
| 收敛速度 | 慢 | 快 |

### 19.2 锚框是几何计算，不是学出来的

每个空间位置的锚框是纯几何生成：

```python
# utils.py:79-97
grid_y, grid_x = torch.meshgrid(...)     # 每个位置的 (x, y) 坐标
grid = (grid + 0.5) / scale              # 归一化到 [0, 1]

wh = 0.05 * (2 ** level)                 # 固定宽高，尺度越大框越大
# level=0: 0.05, level=1: 0.10, level=2: 0.20, level=3: 0.40

proposal = [cx, cy, w, h]                # 每个位置一个锚框
```

和 Faster R-CNN 的 anchor 思路一样——预设固定尺寸，不学习。

### 19.3 关键区别：锚框一样，但打分的特征不同

| | Faster R-CNN | Grounding DINO 两阶段 |
|---|---|---|
| 锚框来源 | 每个位置 9 个 anchor | 每个位置 1 个固定尺寸 |
| 特征来源 | CNN backbone 直接输出 | **Encoder 处理后的特征**（含 self-attention + BiAttention 图文融合） |
| 筛选方式 | RPN 二分类 | ContrastiveEmbed 文本对比分数 |
| 精选后 | RoI Pooling | 作为 Decoder 初始参考点 |

Encoder 输出的特征已经过了 6 层 deformable self-attention + BiAttention，每个位置不再是局部像素，而是融合了上下文和文本语义的全局感知特征。即使锚框很简单，特征里已包含足够信息判断"这个位置大概有没有要找的东西"。

### 19.4 两阶段完整流程

```
Encoder memory: [B, 19797, 256]
    │
    ▼ ① 几何生成锚框
gen_encoder_output_proposals() → [B, 19797, 4]  (每位置固定尺寸)
    │
    ▼ ② enc_out_bbox_embed 预测偏移量
delta = MLP(output_memory) → refined_box = delta + anchor
    │
    ▼ ③ enc_out_class_embed 用文本对比打分
score = ContrastiveEmbed(output_memory, text) → [B, 19797, L]
topk_score = score.max(-1) → 每位置最高置信度
    │
    ▼ ④ 选 Top-K（900 个）
torch.topk(topk_score, 900) → 取最可能的 900 个位置
    │
    ▼ ⑤ 这 900 个作为 Decoder 初始 query 和参考点
    → Decoder 在此基础上精细化
```

### 19.5 设计来源

不是 Grounding DINO 独创，是从 DINO（2022, Zhang et al.）继承的设计。核心洞察：Encoder 已经做了足够多的特征增强，从 Encoder 特征中选 Top-K 位置作为 Decoder 初始 query，比随机初始化收敛更快、效果更好。
