# FCOS3D 深度学习笔记

## 1. 模型概述

FCOS3D 是一种**单目相机 3D 目标检测**模型（论文: [FCOS3D](https://arxiv.org/abs/2104.10956)），将 2D anchor-free 检测器 FCOS 扩展到 3D 空间。输入单张相机图像，输出图像中物体在相机坐标系下的 3D 边框。

- **支持数据集**: nuScenes（10类）、KITTI
- **预训练权重**: 有，位于 `configs/fcos3d/metafile.yml`
- **定位**: nuScenes 3D 检测榜单 Camera-only 赛道的基础模型，PGD 等方法直接继承其架构

---

## 2. 核心设计哲学：将相机内参与网络学习解耦

FCOS3D 最核心的设计是**让神经网络不直接感知相机内参 `cam2img`**，而是通过前/后处理来桥接 2D 图像和 3D 空间。这种"编解码"范式带来两个好处：

1. 网络学习的是**与相机无关的几何表示**，泛化能力更强
2. 同一个模型可以处理不同相机参数的数据，无需重新训练

具体来说：
- **网络本身**：纯 CNN，输入只有图像像素，输出是 2.5D 编码（像素坐标 + 深度 + 尺寸 + 朝向等）
- **相机内参 `cam2img`**：在前处理中将 3D GT **投影**为 2.5D 编码，在后处理中将预测的 2.5D 编码**反投影**回 3D 空间

---

## 3. 模型结构

```

      │
      ▼

│  Backbone: ResNet101 + DCNv2        │
│  - 4个 stage，输出 C2~C5            │
│  - stage3/4 ·  │

│  - 输出通道: [256, 512, 1024, 2048] │
└─────────────────────────────────────┘
      │
      ▼
┌─────────────────────────────────────┐
│  Neck: FPN (Feature Pyramid Network)│
│  - 输入: C2~C5                      │
│  - 输出: P3~P7 共 5 层              │
│  - channels: 256                    │
│  - strides: [8, 16, 32, 64, 128]   │
└─────────────────────────────────────┘
      │ (5 个多尺度特征图)
      ▼
┌─────────────────────────────────────┐
│  Head: FCOSMono3DHead               │
│  (anchor-free, per-pixel 预测)      │
│                                      │
│  共享卷积塔 (stacked_convs=2):       │
│  ├─ cls_convs (分类塔)              │
│  └─ reg_convs (回归塔)              │
│                                      │
│  分类预测: cls_score (C=10)          │
│                                      │
│  回归预测 (5个分支, 各由独立子塔处理): │
│  ├─ offset (2): Δx, Δy 像素偏移     │
│  ├─ depth  (1): 深度 z              │

│  ├─ rot    (1): 朝向角 sin(θ)       │
│  └─ velo   (2): 速度 vx, vy         │
│  → concat → bbox_pred (9)           │
│                                      │
│  辅助预测:                           │
│  ├─ centerness (1): 中心度           │
│  ├─ dir_cls    (2): 方向二分类       │
│  └─ attr_pred  (9): 属性分类        │
└─────────────────────────────────────┘
```

## 4. 训练流程：编码 → 前向 → 损失

训练时，3D GT 先通过相机内参**编码**为网络可学习的 2.5D 表示，网络在编码空间直接与之对比计算损失，无需在线解码回 3D。

### 4.1 前处理 — GT 的"编码" (使用 cam2img 投影)

**数据 pipeline**（`fcos3d_r101-caffe-dcn_fpn_head-gn_8xb2-1x_nus-mono3d.py:19-37`）：

```
nuScenes 样本 (图像 + LiDAR标注 + 相机标定)
      │
      ▼
┌──────────────────────────────────────────┐
│  LoadImageFromFileMono3D                 │
│  加载单张相机图像 → img (H, W, 3)        │
│  提取 cam2img, lidar2cam 等相机参数到meta │
└──────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────┐
│  LoadAnnotations3D                       │
│  加载 GT 标注:                           │
│  ├─ gt_bboxes:     (M, 4)  2D bboxes   │
│  ├─ gt_bboxes_3d:  (M, 9)  3D bboxes   │ ← [x,y,z,w,l,h,yaw,vx,vy]
│  ├─ gt_labels_3d:  (M,)    3D 类别标签  │
│  ├─ attr_labels:   (M,)    属性标签     │
│  └─ ✅ 用 cam2img 投影 3D中心→2D图像:   │
│      centers_2d:   (M, 2)  投影坐标(u,v)│ ← 关键编码!
│      depths:       (M,)    深度值 z     │ ← 关键编码!
└──────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────┐
│  Resize (1600×900) + RandomFlip3D        │
│  图像增强，同步更新 bboxes & centers_2d   │
└──────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────┐
│  Det3DDataPreprocessor                   │
│  BGR→RGB, Normalize, Pad → (N,3,H,W)    │
└──────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────┐
│  Pack3DDetInputs → 打包为:              │
│  inputs = {'imgs': Tensor(N,3,H,W)}      │
│  data_samples.gt_instances_3d:           │
│    bboxes_3d, labels_3d,                │
│    centers_2d, depths, attr_labels      │
│  data_samples.metainfo: cam2img, ...    │
└──────────────────────────────────────────┘
```

**关键点**：`centers_2d` 和 `depths` 是由 `LoadAnnotations3D` 在数据加载阶段，通过 `cam2img` 将 3D GT 中心 `(x, y, z)` 投影到图像平面得到的。这一步之后，GT 就被表示成了网络可以直接回归的 2.5D 形式。

### 4.2 模型前向 — 纯视觉感知（不使用 cam2img）

**代码**: `FCOSMono3D.extract_feat()` + `FCOSMono3DHead.forward()` 

```
输入: imgs (N, 3, H, W)   ← 只有图像像素

Backbone (ResNet101+DCNv2) → C2~C5
Neck (FPN) → P3~P7, 5层, channels=256, strides=[8,16,32,64,128]

Head: 对每层 FPN 特征图并行执行 forward_single()
  ├─ cls_score   (N, 10, Hi, Wi)  — 类别分类
  ├─ bbox_pred   (N, 9,  Hi, Wi)  — 9维回归 (与GT编码格式一致)
  ├─ dir_cls     (N, 2,  Hi, Wi)  — 方向二分类
  ├─ attr_pred   (N, 9,  Hi, Wi)  — 属性分类
  └─ centerness  (N, 1,  Hi, Wi)  — 中心度

输出: 5 组预测，每组对应一个 FPN 层，格式与 GT 编码完全对齐
```

**完全不使用 cam2img**。网络的输入只有图像像素，它学习的是从像素到 2.5D 编码的映射。

### 4.3 损失计算 — 编码空间内的直接对比

**代码**: `FCOSMono3DHead.loss_by_feat()` (`fcos_mono3d_head.py:269-482`)

训练时 Loss 在**编码空间**直接计算，无需将预测解码回 3D，也无需 `cam2img` 参与：

```
网络预测(2.5D编码) ←→ GT(2.5D编码，由前处理"投影"得到)
         │
         ▼ 直接在编码空间对比
      loss_dict
```

**正负样本分配** (`_get_target_single()`, `fcos_mono3d_head.py:845-958`)：

```
每个 FPN 层的每个像素点 → 匹配最佳 GT

条件:
  ① Center Sampling: 像素必须在 GT center_2d 周围 (半径 = 1.5 × stride)
  ② Regress Range: 当前 FPN 层负责特定尺度范围
     stride=8:   [0, 48)      ← 小物体
     stride=16:  [48, 96)
     stride=32:  [96, 192)
     stride=64:  [192, 384)
     stride=128: [384, INF)   ← 大物体
  ③ 取距离最近的 GT (处理一个像素匹配多个 GT 的情况)

满足条件 → 正样本 (计算回归 loss)
不满足   → 负样本/背景 (仅计算分类 loss)
```

**构建回归目标** (`_get_target_single()`):

```python
# 对每个正样本像素 (x_grid, y_grid), 匹配到 GT 后构建目标:
delta_x  = x_grid - center_2d_x     # 像素到投影中心的 2D 偏移
delta_y  = y_grid - center_2d_y
depth    = GT_depth                 # 3D 中心深度
bbox_targets_3d = [delta_x, delta_y, depth, GT_w, GT_l, GT_h, sin(GT_yaw), GT_vx, GT_vy]
centerness_target = exp(-alpha * dist / stride)  # 基于距离的中心度
```

**各损失一览**：

| Loss 名 | 任务 | 损失函数 | 权重机制 | 权重值 |
|---------|------|----------|----------|--------|
| `loss_cls` | 类别分类 (10类) | Focal Loss (γ=2.0, α=0.25) | loss_weight | 1.0 |
| `loss_offset` | 2D 偏移 (Δx,Δy) | Smooth L1 (β=1/9) | code_weight | 1.0 |
| `loss_depth` | 深度 z | Smooth L1 (β=1/9) | code_weight | **0.2** |
| `loss_size` | 3D 尺寸 (w,l,h) | Smooth L1 (β=1/9) | code_weight | 1.0 |
| `loss_rotsin` | 朝向 sin(θ) | Smooth L1 (β=1/9) + sin差分 | code_weight | 1.0 |
| `loss_velo` | 速度 (vx,vy) | Smooth L1 (β=1/9) | code_weight | **0.05** |
| `loss_centerness` | 中心度 | BCE 交叉熵 | loss_weight | 1.0 |
| `loss_dir` | 方向二分类 (0/π) | 交叉熵 | loss_weight | 1.0 |
| `loss_attr` | 属性分类 (9类) | 交叉熵 | loss_weight | 1.0 |

**权重设计解读**：
- `depth` 权重 0.2 — 单目深度估计高度不确定，降权防止其噪声主导梯度
- `velo` 权重 0.05 — 速度预测同样极具挑战性，且 nuScenes 评分中速度指标权重本身就低
- 所有 `loss_weight` 默认 1.0，真正的差异化由 `code_weight` 实现

### 4.4 如何汇总 — 联合训练，一次反向传播

**FCOS3D 采用多任务联合训练，不是各自独立优化。**

```
loss_dict = {
    'loss_cls': ..., 'loss_offset': ..., 'loss_depth': ...,
    'loss_size': ..., 'loss_rotsin': ..., 'loss_velo': ...,
    'loss_centerness': ..., 'loss_dir': ..., 'loss_attr': ...
}
      │
      ▼ MMEngine 自动处理
total_loss = sum(loss_dict.values())   ← 所有 loss 直接求和
total_loss.backward()                  ← 一次反向传播
optimizer.step()                       ← 一次参数更新，更新所有参数
```

所有 9 个 loss 共享同一个 backbone + neck 的梯度信号，端到端训练。code_weight 通过 `bbox_weights` 直接乘到每个样本每个维度的 Smooth L1 loss 上（`fcos_mono3d_head.py:381-385`）：

```python
bbox_weights = ones(N_pos, 9)
bbox_weights = bbox_weights * tensor(code_weight)  # [1, 1, 0.2, 1, 1, 1, 1, 0.05, 0.05]

loss_offset = SmoothL1(pred[:, :2], target[:, :2], weight=bbox_weights[:, :2])
loss_depth  = SmoothL1(pred[:, 2],  target[:, 2],  weight=bbox_weights[:, 2])
# ...

# 分类损失在所有像素上计算 (正+负), avg_factor = num_pos + num_imgs
# 回归损失只在正样本上计算, avg_factor = num_pos
```

## 5. 推理流程：前处理 → 模型 → 后处理

### 5.1 前处理 (Preprocessing)

**代码**: `Det3DDataPreprocessor.forward()` → `collate_data()` (`data_preprocessor.py`)

```
输入: 原始图像 (H, W, 3) + meta 信息(cam2img 等)

  1. numpy/tensor → Tensor(3, H, W)
  2. BGR→RGB (bgr_to_rgb=True)
  3. Normalize: (img - mean) / std
     mean=[103.530, 116.280, 123.675], std=[1.0, 1.0, 1.0]
  4. Pad 到 32 的倍数 (右下角补零)

输出: batch_inputs = {'imgs': Tensor(N, 3, H_pad, W_pad)}
      batch_data_samples (cam2img 等 meta 透传)
```

不涉及 cam2img 计算，仅透传 meta。

### 5.2 模型前向 (同训练，不使用 cam2img)

与训练时的 forward 完全一致，输出 2.5D 编码预测。

### 5.3 后处理 — 预测的"解码" (使用 cam2img 反投影)

**代码**: `FCOSMono3DHead.predict_by_feat()` → `_predict_by_feat_single()` (`fcos_mono3d_head.py:484-715`)

```
输入: 5层预测编码 + batch_img_metas (含 cam2img)

Step 1: 解码到图像空间
  ├─ sigmoid(cls_score), sigmoid(centerness)     → 概率值
  ├─ scale_depth(.)exp(), scale_size(.)exp()      → 正值深度/尺寸
  └─ offset × stride → 恢复到原图尺度
  → bbox_pred = [u, v, depth, w, l, h, sinθ, vx, vy]

Step 2: 计算 2D 中心位置
  └─ bbox_pred[:, :2] = grid_points - offset → (u, v) 图像坐标

Step 3: ✅ 反投影到相机坐标系
  └─ points_img2cam([u,v,depth], cam2img)
     X = (u - c_u) * z / f_u
     Y = (v - c_v) * z / f_v
     Z = z

Step 4: ✅ 局部 yaw → 全局 yaw
  └─ decode_yaw(bbox, centers2d, dir_cls, cam2img)
     方向二分类消除 θ vs θ+π 歧义
     加入射线方向: yaw = atan2(u-c_u, f_u) + dir_rot

Step 5: 3D NMS → 最终输出
  └─ nms_score = cls_score × centerness
     box3d_multiclass_nms → Top-K

输出: pred_instances_3d
  ├─ bboxes_3d: (K, 9) [x, y, z, w, l, h, yaw, vx, vy] (相机坐标系)
  ├─ scores_3d: (K,)
  └─ labels_3d: (K,)
```

## 6. 完整流程总结

```
┌──────────────────────────────────────────────────────────────────┐
│  训练阶段                                                         │
│                                                                   │
│  [3D GT] --cam2img投影--> [GT编码: centers_2d, depths, ...]     │
│                              │                                    │
│  [图像] --> CNN网络 --> [预测编码] --> {Loss 对比} --> total_loss │
│                                            │                      │
│                                    一次 backward 更新所有参数      │
│                                                                   │
│  ✅ cam2img 在"投影"步骤使用，网络前向和Loss计算不使用            │
├──────────────────────────────────────────────────────────────────┤
│  推理阶段                                                         │
│                                                                   │
│  [图像] --> CNN网络 --> [预测编码] --cam2img反投影--> [3D检测结果] │
│                                                                   │
│  ✅ cam2img 在"反投影"步骤使用，网络前向不使用                    │
└──────────────────────────────────────────────────────────────────┘
```

**核心范式**：前处理用 `cam2img` **投影** GT → 2.5D 编码 → 网络学习该编码 → 后处理用 `cam2img` **反投影**回 3D。网络本身始终不感知相机参数。

## 7. 相机参数说明

只需要 **相机内参 `cam2img`**（不需要外参）：

```
cam2img = [[f_u, 0,   c_u],
           [0,   f_v, c_v],
           [0,   0,   1  ]]
```

- `f_u, f_v`: 焦距（像素单位）
- `c_u, c_v`: 光心坐标

存储位置：`img_meta['cam2img']`，由 `Pack3DDetInputs` 自动包装。

## 8. 关键设计细节

### 8.1 Centerness（中心度）

预测每个像素到 GT 中心的归一化距离，用于抑制远离物体中心的低质量预测：

```python
centerness_target = exp(-centerness_alpha * dist / stride)  # alpha=2.5
```

代码: `fcos_mono3d_head.py:955`

### 8.2 方向分类器 (Direction Classifier)

朝向角 θ 有周期性歧义（θ 和 θ+π 在单目图像中看起来相似），引入二分类器：
- bin 0 / bin 1: 朝向在 [0, π) 或 [π, 2π)
- `dir_offset = π/4` 让 bin 边界避开车头正对相机的情况

代码: `fcos_mono3d_head.py:232-267`

### 8.3 sin 差分 (diff_rad_by_sin)

计算旋转 loss 时用 sin 差而非直接角度差，避免角度周期性问题：

```python
sin_diff = sin(θ_pred) * cos(θ_gt) - cos(θ_pred) * sin(θ_gt)
```

代码: `fcos_mono3d_head.py:207-230`

### 8.4 Center Sampling

只有落入 GT 中心附近（`center_sample_radius=1.5 × stride`）的像素才被分配为正样本，减少物体边缘处的不确定性。

### 8.5 可学习 Scale 参数

每层 FPN 有独立可学习 scale 参数 (`Scale(1.0)`)，用于自动调整 offset、depth、size 预测的尺度。

## 12. DCNv2（可变形卷积网络 v2）技术详解

> DCNv2 在 FCOS3D 中作为 Backbone（ResNet101）的增强组件使用（`ResNet101 + DCNv2`），将 stage3 和 stage4 中的部分普通卷积替换为可变形卷积，提升对目标形状变化和遮挡的建模能力。

### 12.1 普通卷积的问题

传统 CNN 中，卷积核（如 3×3）采用**固定规则网格采样**，每次都在固定位置取值：

```
普通 3×3 卷积采样点（固定网格）:
    (-1,-1)  (-1,0)  (-1,1)
    (0,-1)   (0,0)   (0,1)
    (1,-1)   (1,0)   (1,1)
```

👉 现实中的物体会旋转、变形，不一定刚好对齐网格 → **普通卷积不够灵活**

### 12.2 DCN v1：让卷积采样点"会动"

DCN（Deformable Convolutional Networks）核心思想：

👉 **为每个采样点学习一个 2D 偏移量 Δp**

```
采样位置 = 原固定位置 + 可学习偏移量 (Δx, Δy)
```

📌 效果：卷积核可以"变形"，自动对齐目标边缘或关键部位。

### 12.3 DCNv2 的关键改进：Modulation（调制权重）

DCNv2 在 v1 基础上增加了一个重要机制：

👉 **每个采样点不仅有位置偏移 Δp，还有调制权重 Δm（0~1）**

公式：

```
y(p₀) = Σ w(pₙ) · x(p₀ + pₙ + Δpₙ) · Δmₙ
```

其中：
- `p₀`：像素中心位置
- `pₙ`：固定的初始网格偏移（如 (-1,-1), (-1,0), ...）
- `Δpₙ`：**可学习的位置偏移**（让卷积核变形）
- `Δmₙ`：**可学习的调制权重**（控制每个采样点的贡献度）
- `w(pₙ)`：卷积核权重

### 12.4 DCNv2 本质做了两件事

总结成一句话：

👉 **不仅"采哪里"可以学，还"采多少"也可以学**

| 机制 | v1 | v2 | 作用 |
|------|:--:|:--:|------|
| Offset（位置偏移）| ✅ | ✅ | 让卷积核形状可变，对齐目标结构 |
| Modulation（强度调制）| ❌ | ✅ | 控制每个采样点重要性，抑制无关区域/背景噪声 |

### 12.5 为什么需要 Modulation？

研究发现，DCNv1 虽然能对齐目标，但**会"看太多不相关区域"**（采样点可能漂移到背景上）。

DCNv2 通过 modulation 机制：
- **更关注目标区域** — 目标上的采样点获得高权重
- **少看背景噪声** — 无关区域的采样点权重被压制到接近 0

👉 这本质上相当于在可变形卷积中嵌入了**空间注意力机制**。

### 12.6 直观类比

| 方法 | 类比 |
|------|------|
| 普通卷积 | 用**固定模板**盖章，每次盖在同样位置 |
| DCN v1 | 模板可以**拉伸/扭曲**去贴合目标形状 |
| DCNv2 | 不仅能扭曲，还能**决定哪些地方用力按，哪些地方轻轻带过** |

### 12.7 在 FCOS3D 中的应用

```python
# mmdet3d 中使用 DCNv2 的典型配置（MODEL 中指定）：
backbone=dict(
    type='ResNet',
    depth=101,
    dcn=dict(
        type='DCNv2',           # 可变形卷积 v2
        deform_groups=1,        # 控制 offset 和 modulation 的分组数
        fallback_on_stride=False # stride=1 时才应用 DCN
    ),
    stage_with_dcn=(False, False, True, True),  # 只在 stage3/4 使用 DCNv2
)
```

设计考量：
- **stage1/2 不用 DCNv2**：浅层特征分辨率高，计算量大，且浅层纹理特征不需要太强的几何变形
- **stage3/4 使用 DCNv2**：高层语义特征分辨率低，DCNv2 开销可接受，且高层特征更受益于灵活的几何建模

### 12.8 效果

DCNv2 在多个任务中显著提升性能：
- ✅ 目标检测（Faster R-CNN / FCOS3D）
- ✅ 实例分割（Mask R-CNN）
- ✅ 语义分割

👉 根本原因：**更强的几何建模能力 + 更灵活的特征提取**

### 12.9 一句话总结

> **DCNv2 = 可变形卷积（v1）+ 空间注意力权重（Modulation）**
>
> 让 CNN 更聪明地"看哪里"（offset），更合理地"看多少"（modulation）。

## 13. 多任务头（Multi-Task Head）设计原理

> FCOS3D 的 Head 是一个典型的多任务头：从同一个 FPN 特征出发，分支出分类、offset、depth、size、rot、velo 等多个预测。本节深入解释"一个共享特征向量为什么能同时服务多个任务"。

### 13.1 核心结论

👉 **一个共享特征向量可以同时服务多个任务头**

本质原因：

> **高维特征可以被不同任务头"投影/解读"为不同信息**

### 13.2 整体结构范式

```
Backbone / Neck → 共享特征 f (channel=256)
                     │
          ┌──────────┼──────────┬──────────┬──────────┐
          ▼          ▼          ▼          ▼          ▼
       cls塔      offset塔   depth塔    size塔     rot塔  ...
          │          │          │          │          │
          ▼          ▼          ▼          ▼          ▼
      cls_score   offset    depth     size      rot_sin   ...
    (分类子空间) (位置子空间)(深度子空间)(尺寸子空间)(朝向子空间)
```

FCOS3D 的具体分支：

| 分支 | 输出维度 | 含义 |
|------|---------|------|
| `cls_score` | C=10 | 类别分类 |
| `offset` | 2 | 像素偏移 Δx, Δy |
| `depth` | 1 | 深度 z |
| `size` | 3 | 3D 尺寸 w, l, h |
| `rot` | 1 | 朝向角 sin(θ) |
| `velo` | 2 | 速度 vx, vy |

👉 所有分支共享同一个 FPN 特征图，属于**典型 3D 检测 multi-head 设计**。

### 13.3 为什么"一个向量可以多用"

#### ① 高维表示能力强

一个特征向量 `f ∈ ℝ²⁵⁶` 可以同时编码多种信息：
- **语义**（是什么：车、人、障碍物）
- **几何**（在哪、多大、什么朝向）
- **动态**（往哪动、多快）

不同信息在高维空间中可**近似解耦**，互不干扰。

#### ② 任务本质是同一目标的不同属性

所有 head 都是在描述**同一个物体**：

| 任务 | 回答的问题 |
|------|-----------|
| cls | 这是什么？ |
| offset | 它在图像中精确在哪？ |
| depth | 它离我多远？ |
| size | 它有多大？ |
| rot | 它朝哪个方向？ |
| velo | 它在怎么动？ |

👉 这些属性天然相关（比如"车"的 size 和"行人"的 size 分布完全不同），所以共享特征是合理的，甚至能互相促进。

#### ③ 训练会自动形成"子空间分工"

不同 loss 驱动不同维度的学习：

- **分类 loss** → 学习语义区分
- **depth loss** → 学习空间深度
- **velocity loss** → 学习动态信息

最终形成隐式结构：

> **一个向量 = 多种信息的组合，各任务读取各自关心的子空间**

### 13.4 任务头在做什么（数学本质）

#### 线性视角（简化理解）

每个任务头本质上是一个投影：

```
y_task = W_task · f + b
```

👉 等价于：**把 f 投影到该任务相关的子空间**

#### 实际结构（更强）

不是简单线性，而是：

```
f → Conv/MLP → 非线性变换 → 输出
```

FCOS3D 中每个分支有独立的**卷积子塔**（`stacked_convs=2`），即 2 层 Conv + GN + ReLU：

```python
# 以 offset 分支为例（简化）：
# 共享卷积塔提取通用特征后，各分支再经过自己的子塔
offset_feat = offset_convs(shared_feat)   # 2层 conv，独立参数
offset_pred = offset_conv(offset_feat)     # 1×1 conv 输出 2 通道
```

👉 本质：**非线性投影 + 特征重组**，远强于单层线性映射。

### 13.5 为什么要"独立子塔"（而不是完全共享）

如果所有任务直接共享同一个输出头：

❌ **任务冲突（Task Conflict）**

不同任务关注不同性质的信息：

| 任务类型 | 关注点 | 特征偏好 |
|---------|--------|---------|
| 分类 | 语义 | 纹理、全局上下文 |
| 回归（offset/size） | 边界/几何 | 边缘、局部细节 |
| 深度 | 空间结构 | 透视线索、尺度 |

这些任务的梯度方向可能**冲突**，导致训练不稳定。

#### ✅ 解决方案：Task-Specific Tower

```
共享特征（信息融合，效率高）
   │
   ├─→ 分类子塔（专注语义）
   ├─→ 回归子塔（专注几何）
   ├─→ 深度子塔（专注空间）
   └─→ ...
```

好处：
- **减少梯度冲突** — 各任务的梯度通过独立路径传播
- **提升任务专注度** — 每个子塔只学自己关心的表示
- **提高整体性能** — 共享 + 解耦取得最佳平衡

### 13.6 与大语言模型的类比

LLM 预测下一个词：

```
h → W_vocab · h → softmax → 下一个词
```

本质也是：**一个向量 → 投影到某个任务空间**

多任务检测头：

```
h → W₁ → 任务1（分类）
h → W₂ → 任务2（深度）
h → W₃ → 任务3（尺寸）
```

👉 二者在数学上完全一致：**同一个隐藏表示被不同投影矩阵映射到不同输出空间**。

### 13.7 设计多任务头的关键原则

| 原则 | 说明 | FCOS3D 的做法 |
|------|------|--------------|
| **共享 + 解耦** | 前面共享（效率+融合），后面分支（减少冲突）| shared_convs(2层) → 各分支独立子塔 |
| **特征维度足够** | ≥64/128/256 channels | channel=256 |
| **子塔有表达能力** | 不能只用单层 Linear | stacked_convs=2（Conv+GN+ReLU）|
| **Loss 平衡** | 不同任务 scale 不同，需加权 | code_weight: depth×0.2, velo×0.05 |
| **表达方式合理** | 输出编码要适配任务特性 | rot→sin(θ), size→(w,l,h) 对数空间 |

### 13.8 一句话总结

> **多任务头 = 用多个"投影器"从同一个高维表示中读取不同信息**

更直白：

> **一个向量是"压缩的世界"，多个任务头是在"不同角度解读它"**

这种设计在 3D 检测、2D 检测、多任务 NLP 中都是核心范式 — 理解它就理解了一半的现代检测头设计。

## 14. 多任务损失设计：分类 + 回归

> 本节系统拆解 FCOS3D 中分类损失和 5 个回归分支的 loss 选型原理、计算范围（正样本/负样本）、以及 loss 平衡策略。

### 14.1 总损失结构

多任务检测的总 loss 是各任务 loss 的加权和：

```
L_total = L_cls + λ₁·L_offset + λ₂·L_depth + λ₃·L_size + λ₄·L_rot + λ₅·L_velo
```

> **每个任务一个 loss，加权求和**

### 14.2 分类损失（Classification Loss）

`cls_score (C=10)` — 每个像素点的类别预测。

#### ① Cross Entropy（最基础）

```
L_cls = -Σ y·log(p)
```

适用于单标签分类。缺点：检测中负样本远多于正样本，会导致 loss 被简单负样本主导。

#### ② Focal Loss（检测中最常用 🔥）

```
L_cls = -α·(1-p)^γ·log(p)
```

| 参数 | 典型值 | 作用 |
|------|--------|------|
| γ | 2.0 | 抑制容易样本的 loss 贡献，让模型更关注难样本 |
| α | 0.25 | 平衡正负样本（正样本少 → 提高正样本权重） |

👉 FCOS3D 使用的就是 Focal Loss（γ=2.0, α=0.25）

### 14.3 回归损失（Regression Loss）— 5 个分支

#### ① offset（Δx, Δy）— 2D 位置偏移

```
L_offset = Smooth L1(pred - target)
```

数值范围在像素级别，使用 Smooth L1（β=1/9）直接回归。

#### ② depth（z）— 深度

```
L_depth = Smooth L1(pred - target)
```

两种常见设计：

| 方式 | 公式 | 特点 |
|------|------|------|
| 直接回归 | `|z_pred - z_gt|` | 简单，但远距离误差大 |
| log 空间 🔥 | `|log(z_pred) - log(z_gt)|` | 远距离更容忍，近距离更精确 |

FCOS3D 采用 **log 空间 + exp 解码**：网络预测 `log(z)`，loss 在 log 空间计算，推理时 `z = exp(pred)`。

👉 原因：深度范围跨度大（几米到上百米），log 空间让不同距离的量级更均匀。

#### ③ size（w, l, h）— 3D 尺寸

```
L_size = Smooth L1(pred - target)
```

同样在 **log 空间** 计算，推理时 `exp` 解码。原因：
- 不同类别尺寸差异大（行人 0.5m vs 卡车 10m+）
- log 空间归一化了量级差异

#### ④ rotation — 朝向角

FCOS3D 使用 **sin(θ) 编码**：

```
L_rot = Smooth L1(sin(θ_pred) - sin(θ_gt))
```

**为什么不直接用 θ？**

θ = -π 和 θ = π 是同一朝向，但数值差 2π → 产生**不连续的 loss 跳变** ❌。

**sin(θ) 编码的好处**：
- sin(-π) = sin(π) = 0 → 连续性 ✅
- 配合 sin 差分公式避免周期性问题：
  ```python
  sin_diff = sin(θ_pred)·cos(θ_gt) - cos(θ_pred)·sin(θ_gt)
  ```

有时还会**同时预测 sin(θ) 和 cos(θ)** 以更稳健地恢复角度。

#### ⑤ velocity（vx, vy）— 速度

```
L_velo = Smooth L1(pred - target)
```

nuScenes 特有输出，速度预测极具挑战性（单目缺乏时序信息）。

### 14.4 关键：回归损失只对正样本计算！⚠️

| 损失类型 | 计算范围 | 原因 |
|---------|----------|------|
| **分类 loss** | ✅ 所有样本（正 + 负）| 负样本也需要学会判断为"背景" |
| **回归 loss** | ❌ 只对正样本 | 负样本没有 bbox/目标，无法计算回归 |

实现上通过 mask 机制：

```python
# mask = 1（正样本），mask = 0（负样本/背景）
L_reg = mask · SmoothL1(pred - target)

# avg_factor = num_pos  # 只在正样本上取平均
```

### 14.5 Loss 平衡策略

不同任务的数值量级差异巨大：

| 任务 | 数值范围 | 如果不平衡会怎样 |
|------|---------|---------------|
| offset | 像素（0~100）| — |
| depth | 米（1~100+）| 大数值主导梯度 |
| size | 不同尺度（0.5~10+）| — |
| velocity | m/s（0~30）| — |

👉 如果直接等权求和，**量级大的 loss 会主导训练** ❌

#### FCOS3D 的做法：手动权重（code_weight）

```python
code_weight = [1.0, 1.0,   # offset
               0.2,         # depth  ← 降低，防止深度噪声主导
               1.0, 1.0, 1.0,  # size
               1.0,         # rot
               0.05, 0.05]  # velo   ← 大幅降低

# 权重直接乘到每个样本每个维度的 Smooth L1 loss 上
loss_offset = SmoothL1(pred[:, :2], target[:, :2], weight=bbox_weights[:, :2])
loss_depth  = SmoothL1(pred[:, 2],  target[:, 2],  weight=bbox_weights[:, 2])
# ...
```

权重设计解读：
- **depth × 0.2** — 单目深度估计高度不确定，降权防止其噪声主导梯度
- **velo × 0.05** — 速度预测极具挑战性，且 nuScenes 评分中速度指标权重本身就低
- **offset/size/rot × 1.0** — 相对稳定的回归目标

#### 高级方法（FCOS3D 未使用但值得了解）

| 方法 | 核心思想 |
|------|---------|
| Uncertainty Weighting | 让网络自动学习每个任务的不确定性（`1/σ²`），不确定的任务自动降权 |
| GradNorm | 动态调整权重使各任务梯度量级接近 |
| Dynamic Loss Scaling | 根据训练过程中各 task 的收敛速度调整权重 |

### 14.6 bbox_pred（9 维 concat）的 loss 处理方式

```
bbox_pred = concat[offset(2), depth(1), size(3), rot(1), velo(2)] = 9维
```

#### ✅ 推荐做法：分开算 loss

```python
L = L_offset(bbox_pred[:, 0:2]) + L_depth(bbox_pred[:, 2]) \
  + L_size(bbox_pred[:, 3:6])   + L_rot(bbox_pred[:, 6]) \
  + L_velo(bbox_pred[:, 7:9])
```

👉 每个子任务独立计算，清晰可调 — **FCOS3D 采用此方式**。

#### ❌ 不推荐：直接对整个 9 维向量求 loss

```
L = SmoothL1(bbox_pred, bbox_gt)  # 直接 9 维 loss
```

问题：不同物理意义混在一起，scale 不同，难以优化。

### 14.7 一句话总结

> 分类：**Focal Loss**（解决正负不平衡）→ 在所有样本上计算
>
> 回归：**Smooth L1**（各分支独立）→ 只在正样本上计算
>
> 多任务：**各自独立 loss + code_weight 加权求和** → 一次 backward 端到端更新

## 15. 多任务梯度流动机制

> 核心问题：总 loss 是多个子 loss 的和，反向传播时参数到底接收哪些 loss 的梯度？是各管各的还是混合的？

### 15.1 先给结论

> **不是"每个参数只用自己的 loss"，而是"每个参数接收所有它参与路径上的 loss 的梯度"**

### 15.2 计算图视角

```
共享特征 f (backbone/neck 输出)
   │
   ├─→ θ_cls    → L_cls
   ├─→ θ_offset → L_offset
   ├─→ θ_depth  → L_depth
   ├─→ θ_size   → L_size
   ├─→ θ_rot    → L_rot
   └─→ θ_velo   → L_velo

总 loss: L = L_cls + L_offset + L_depth + L_size + L_rot + L_velo
```

### 15.3 两类参数的梯度截然不同

#### ① Task-Specific Head 参数（θ_cls, θ_depth, ...）

梯度只来自自己的任务：

```
∂L/∂θ_cls   = ∂L_cls/∂θ_cls      ← 只有分类 loss
∂L/∂θ_depth = ∂L_depth/∂θ_depth  ← 只有深度 loss
∂L/∂θ_rot   = ∂L_rot/∂θ_rot      ← 只有旋转 loss
```

✅ **只受自己任务的 loss 影响**，其他 loss 的梯度在计算图上不通向这些参数。

#### ② 共享参数（Backbone / Neck）🔥

梯度是所有任务梯度的叠加：

```
∂L/∂θ_shared =
    ∂L_cls   /∂θ_shared
  + ∂L_offset/∂θ_shared
  + ∂L_depth /∂θ_shared
  + ∂L_size  /∂θ_shared
  + ∂L_rot   /∂θ_shared
  + ∂L_velo  /∂θ_shared
```

> **所有任务的梯度会在共享层"汇合叠加"**

### 15.4 直觉图景：多个任务在"拉"同一个特征

```
每个任务都在"拉"共享特征往自己有利的方向走：

  分类    →  想让特征更"语义化"
  depth   →  想让特征更"几何化"
  size    →  想让特征更"结构化"
  velo    →  想让特征更"动态化"

结果：
  共享特征被多个任务"共同塑形"
```

### 15.5 梯度冲突（Task Conflict）

核心问题：

```
∇L_cls 和 ∇L_depth 可能方向不一致
```

甚至：

```
cos(∇L_cls, ∇L_depth) < 0   → 方向相反！
```

👉 意味着：**一个任务在"帮忙"，另一个在"拆台"**

这也解释了为什么要**共享 + 解耦**的设计：

```
共享 backbone（梯度汇合，会有冲突）
   ↓
task-specific tower（梯度分流，减少冲突）
   ↓
各自任务
```

### 15.6 总结：梯度流向对照表

| 参数类型 | 接收哪些 loss 的梯度 | 
|---------|-------------------|
| θ_cls（分类头）| 只有 L_cls |
| θ_depth（深度头）| 只有 L_depth |
| θ_offset（偏移头）| 只有 L_offset |
| θ_size / θ_rot / θ_velo | 各自只有自己的 loss |
| **θ_shared（backbone/neck）** | **所有 loss 的梯度之和** |

### 15.7 深层理解：与多任务头原理的联结

你之前理解的：

> "一个特征向量 = 压缩的世界，多头 = 不同角度解读"

现在可以从训练角度加深：

> **这个"压缩的世界"不是天然形成的 — 它是多个任务的梯度在训练中共同塑造出来的**

Multi-Task Learning 本质：

```
共享表示 = 多个任务梯度的"折中解"
```

### 15.8 进阶方向

这里会引出几个重要的研究问题：

| 问题 | 方法 |
|------|------|
| 怎么解决梯度冲突？ | PCGrad（投影冲突梯度）、GradNorm |
| 怎么自动调 loss 权重？ | Uncertainty Weighting、Dynamic Balancing |
| 要不要完全共享？ | Shared Backbone vs Task-Specific Backbone 的权衡 |
| 特征解耦？ | 显式拆分 semantic / geometric / dynamic 子空间 |

### 15.9 一句话总结

> **loss 是加在一起的，但梯度是"按计算图分流 → 在共享层汇合"的**
>
> Head 参数各管各，共享参数被所有任务共同塑形 — 这就是 Multi-Task Learning 的全部秘密。
